Endeavor 1 — Events-as-State Architecture

Status: Working document · Phases 0, 1.1–1.4, and 2 complete · Phases 1.5 and 1.6 pending · REPL implementation plan in docs/repl-and-substrates.html · Subagent & handoff research in docs/subagents-research.html · Canonical source: docs/session-design.html

Adopted framing

A session is the persisted state of one Omega instance. It is resumable at "awaiting user" boundaries, against the same Omega version that created it. Subagents are child sessions; everything else is a derived view.

Process incarnation is irrelevant — a session survives restarts and ends when the agent's logical life ends. State at any moment is fold(events); the LLM context, UI feed, and context-hash tree are derived projections. Compaction is a within-session event; forking is a deferred feature (Phase 4). No cross-version replay: the Rust type system + fold invariants reject incompatible events structurally.

Domain state and the consistency chain

The events-as-state promise is precisely observational equivalence after resumption: an external observer cannot distinguish a resumed session from one that never stopped. Two relevant observers:

  • The LLM and the tools. Future turns must unfold the same way.
  • The UI / user. The past must display the same way.

A piece of agent state is domain state iff a difference in it could be observed by either observer. Everything else — connection handles, channels, file descriptors, freshly-recomputed indices — is plumbing: the resumed process needs some technical instance, but the contents need not match the predecessor's.

The contract between domain state, events, and UI is the chain:

domain-relevant ⇒ persisted ⇒ visible in UI

Both implications are load-bearing:

  • domain-relevant ⇒ persisted — anything affecting future behavior or past display must be in the event log. No carve-outs. Concrete consequence: the system prompt is persisted at session start in SessionStartedEvent.system_prompt and stays fixed for the lifetime of the session. Strict resume must use that persisted value — not re-read AGENTS.md from disk, which would introduce a behavioral change with no recorded event. To pick up an updated system prompt, start a new session.
  • persisted ⇒ visible in UI — anything in the event log should have a UI representation. No secret state; no events without UI.
✓ Gate-able criterion

Stop Omega. Start a fresh process from the persisted events. The resulting domain state must match what it would have been right after the last user message. Plumbing fields are not part of this check — by construction they cannot influence behavior or display. The classification of "domain state vs. plumbing" is enforced at compile time (see Phase 2 follow-up below).

Design history — rejected approaches, subagent-vs-fork, session/event refs

Rejected approaches (full rationale: git show 9d311ee)

  • Session = process lifetime — conflates incarnation with identity.
  • Session = LLM-visible context — boundaries leave our control; provider-dependent; ignores non-LLM events.
  • Session = persisted-log lifetime — right answer for wrong reason; subsumed by the process-state framing with a testable criterion.
  • Session = git-style tree of contexts — useful internal index, not identity; forking is now a deferred feature, not a primitive.

Subagent-vs-fork design choices

Session creation takes a seed ([system_prompt, initial_user_message]). Metadata records parent_session_id and an Origin enum (Root / SubagentOf{...}, extendable to ForkOf). Session IDs are flat UUID v7 — never encode parent relationships in IDs. On-disk: subagent sessions in subfolders of the parent; forks would be top-level. Full type definitions and rationale: docs/sessionref-design-proposal.html.

Session and event references

Both sessions and events have globally unique UUID v7 IDs. SessionRef { session_id, event_id: Option<EventId> } — None means "session as a whole"; Some pins a moment. References are by ID; folder layout is a resolution hint only. Every cross-session event must render visibly in the feed with literal IDs and a "more" button from day one — transparency floor, not polish. Full shape, serde conventions, and rationale: docs/sessionref-design-proposal.html. Implemented in commit b8930fc.

Long autonomous runs — retracted concern

Mid-turn resume is intentionally absent. Every realistic failure mid-run is either an infrastructure flake (Omega backs off), a broken tool (surfaces to the agent loop), or an Omega bug (benchmark should fail loudly). Tool-completion checkpoints would mask exactly the cases we want to see fail. Their absence is correct.

Decision: drop strict session resume — ✓ resolved

✓ Strict session resume is being dropped entirely

Reached in the session-resumption discussion that paused Phase 1.2 work (the ⏸ pause was previously recorded in repl-and-substrates.html and is now resolved here). Applies to all sessions, not just REPL.

Reasoning

  • The observational-equivalence promise was always a half-truth. strict_resume preserves the agent's internal state (conversation history, tool definitions, system prompt, feature flags, last user-message boundary) but never preserved the world those tools observe. After a resume: read_file("foo.rs") can return different bytes following a git pull; run_command("cargo test") output drifts with a dependency bump; fetch_url(…) was never stable across time; even list_files(".") sees a changed target/. The closed-system equivalence we deliver is the agent's bookkeeping; the open-system drift was always there. REPL just makes the drift impossible to ignore — the kernel state becomes part of what diverges, not merely the world.
  • The user's empirical usage data does not motivate strict resume. Never used cross-machine; never desired when the repo had moved; the only sustained pattern (weekly Arch reboot) is scheduleable around natural session boundaries. Other cases (IT-forced work-laptop reboot, process-count concerns) are rare and fork-from-last-user-message would be acceptable for them — but the user did not endorse adding fork-from-event either. Idle sessions are cheap; there is no resource pressure that demands resumption.
  • Forensics is real and crucial, but it's AI-based, not UI-based. The forensic interface is a fresh agent reading events.jsonl from a past session directory and reasoning over it — not the user scrolling a past-session UI. This works today via existing read_file / grep_files tools; no new code is required. The schema-evolution-loud rule already guards readability.
  • Future-other-users argument is hedged and the fallback is clean. If/when Omega is ever offered to other users with strict resume as a hard requirement, the right answer is to fork Omega as a base for a slimmed-down agent — not to burden the current codebase indefinitely with weak-guarantee machinery for a use case the primary user does not have.
  • Sunk cost is not a reason to keep code. A non-trivial amount of session_resume.rs was real engineering (Phase 2, the typed domain-state snapshot, the F11 compaction-gap closure). Acknowledged; not a reason to keep code that doesn't earn its complexity going forward.

Consequences for the codebase

  • To be removed (in a backlog follow-up sub-session, see repl-and-substrates backlog): strict_resume(), StrictResumeError and all its variants, read_events_strict, the fold_* reconstruction helpers, DomainSnapshot (the typed-domain-state work documented under oq-allowlist), and round_trip_gate in crates/omega-agent/tests/internal.rs. Any WsMessage variant whose sole purpose was strict resume goes with them.
  • Stays: the event-reading half of session_resume.rs — typed parsing of events.jsonl. AI-based forensics needs it. If a future view-only UI ever materializes, it needs it too.
  • Phase 1.2 simplifies: the tool_selection migration in SessionStartedEvent is still right, but the session_resume.rs touchpoint collapses to (at most) a serde round-trip unit test on the new event field. No fold_tool_selection to write. See repl-and-substrates Next concrete steps.

What rises in priority instead

Live-session intelligibility for REPL. The cold-path "past session" problem dissolves into AI-based forensics. The hot-path "current session" problem — whether the WS stream + frontend surface enough of a live REPL turn (persistent kernel variables, streaming stdout, sh() output, tracebacks, cell boundaries) for the user to keep intuition intact and form new ideas — is where attention belongs. Scoped as a read-only audit sub-session in the repl-and-substrates backlog.

Phase 1.6 WS protocol → SessionId migration ⏳ Pending

⚠ Pending — required before Phase 3

Today's WsMessage::ResumeSession, ::AppendEvent, etc. identify sessions by folder name (a resolution hint, per the framing) instead of by canonical SessionId. With Phase 1's decoupling of folder name from SessionId, the WS protocol now carries the weaker identifier.

Migration steps:

  • Replace session_dir: String with session_id: SessionId in every WS message variant that identifies a session.
  • Establish a canonical SessionId → folder path lookup at the WS boundary. Scan-on-demand is the simplest start; an in-memory registry replaces it if scan latency becomes noticeable (see open question below).
  • Update HttpHandler / router endpoints that accept session paths to accept (or prefer) SessionId.
  • Update the leptos frontend to send SessionId rather than directory strings.

Required before Phase 3 because SubagentSpawned.child.session_id must be navigable end-to-end from the UI.

Open questions — Endeavor 1 scope

✓ Typed domain-state snapshot — implemented (Phase 2 follow-up + feature-flags correction); now scheduled for removal

Status update: the work below was delivered as described and remains in session_resume.rs today, but with the decision to drop strict resume, DomainSnapshot and the fold_* helpers no longer have a purpose and are scheduled for removal along with strict_resume(). The principle recorded at the bottom of this callout — any mechanism that mutates domain state must emit a domain event — outlives the implementation and stays valid.

Completed in the Phase 2 follow-up sub-session, with a correction applied in a subsequent sub-session. The delivered items were:

  • DomainSnapshot struct in crates/omega-agent/src/session_resume.rs — typed, PartialEq, with doc comment explaining the compile-time classification contract.
  • Agent::domain_snapshot() — exhaustive let Self { … } = self; destructuring with no .. rest pattern; every plumbing field carries an inline domain-terms comment.
  • Strict resume now uses SessionStartedEvent.system_prompt rather than re-reading AGENTS.md from disk — closing the domain-relevant ⇒ persisted carve-out. Failing loudly (StrictResumeError::MissingSystemPrompt) when the field is absent per the schema-evolution-loud rule.
  • Feature flags (FeatureFlags) are domain state, not plumbing. They determine which tools are exposed to the LLM on every call — a difference is observable in future turns. Added features: FeatureFlags to DomainSnapshot; fold_features recovers them from the first SessionStartedEvent in events.jsonl on strict resume; fails loudly (StrictResumeError::MissingFeatures) when the event is absent.
  • round_trip_gate exercises the features round-trip with non-default flags (repl: true, subagents: true); a single assert_eq! on two DomainSnapshots covers all domain fields including features.

Principle recorded: any mechanism that mutates domain state must emit a corresponding domain event. For feature flags, no mutation mechanism exists today (they are read from env at process start and never changed), so the immediate work is only restoration on strict resume. If a future toggle is added (admin WS message, auto-disable on failures, etc.), it must come with a FeatureFlagsChanged domain event — structurally analogous to ModelChanged / EffortChanged.

One new carve-out was surfaced during the field audit — see oq-cwd-carveout below.

✓ [oq-cwd-carveout] config.cwd latent carve-out — moot

Was a follow-up flagged during the Phase 2 audit: config.cwd was not persisted in any event but would affect tool-call behaviour (path resolution, AGENTS.md discovery) after a strict resume with a different cwd.

Now moot with the decision to drop strict resume: there is no resume path against which the silent semantic divergence could materialize. The principle still applies if any future feature reconstructs an Agent from persisted state — reopen if that happens.

✓ [oq-repl-view] View-only WS endpoint for REPL session forensics — deferred indefinitely

Was proposed as a way to preserve UI-based browsing of past REPL sessions once strict_resume started refusing them.

Deferred indefinitely following the decision to drop strict resume. The actual forensic workflow is AI-based: a fresh agent reading events.jsonl from a past session directory. A UI scrollback of past events would be a redundant second path — useful only if a concrete UX need surfaces that the AI cannot serve. Reopen if such a need appears.

✓ [oq-repl-replay] Strict-resume replay for REPL sessions — moot

Was proposed as the principled fix to REPL strict-resume divergence: replay historical python_repl calls into a fresh subprocess on resume, with a policy for divergence between recorded and replayed output.

Moot with the decision to drop strict resume. The non-determinism worry that motivated it generalizes beyond REPL (read_file, run_command, fetch_url all drift across resumes), which was itself part of the case for dropping strict resume rather than papering over it.

⚠ Registry vs. scan for SessionId → folder resolution — needed for 1.6

Phase 1.6 needs a canonical lookup from SessionId to folder path. Scan-on-demand is the simplest start; a small in-memory registry can replace it if scan latency becomes noticeable. Performance-driven; deferrable to 1.6 execution time.

✓ All 9 SessionRef-shape questions — resolved in Phase 1 design

Full rationale, type definitions, and sample JSON in docs/sessionref-design-proposal.html (frozen reference). Summary: newtypes, one SessionRef type, struct serde with camelCase, <sid>#<eid> display format, Hash+Eq only (no Ord), omega-types crate, envelope placement, PK/FK usage rule, Origin with #[non_exhaustive].

✓ Crash-recovery UX — resolved: silent abandon

On resume after a crash mid-turn, the partial turn is silently discarded and state is restored to "just after last user message." No prompt is shown. The TurnInterrupted event in the stream is sufficient forensic signal.
