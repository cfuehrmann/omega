REPL, Subagents & Handoff Automation — Research Doc

Status: Research document — not a proposal toward imminent implementation · Supersedes docs/rlm-exploration-proposal.html · Endeavor 1 (events-as-state) work is tracked in docs/session-design.html · Canonical path: docs/repl-and-subagents-research.html

§1 Framing

§1.1 Research-mode stance

⚠ This is a research document, not a proposal

The question of whether and how to add a REPL, vanilla subagents, RLM-style recursive subagents, or automated handoffs to Omega is open. None of these are committed work. This document collects evidence, plants research angles, and identifies the open questions that would need answers before any direction is chosen.

Hard-data bar: a direction earns committed work only when there is empirical evidence — from Harbor sweeps, from instrumented everyday use, or from the academic literature applied to Omega's specific task profile — that it improves outcomes in the metrics that matter (gate passage, tokens/commit, commit quality, time/commit). Intuition and analogies from other agents do not clear the bar.

✓ [b0bc06] Feature flags gate everything

Any capability added under these research directions ships behind two independent runtime flags so Harbor can sweep all four combinations without rebuilds:

subagents=off subagents=on
repl=off baseline (current Omega) vanilla subagents
repl=on REPL only (CodeAct-style) RLM (REPL + llm_query)

RQ-D (handoff automation) does not fit this matrix; it is governed by a separate flag or by the seeded-handoff workflow itself (see §1.5 and §6).

§1.2 What Omega specifically brings

Most multi-agent research is done on benchmark datasets with one-shot evaluations. Omega offers a materially different substrate:

  • The user's daily use as a continuous experiment. Every coding session is a real-world data point: real problems, real codebases, real time pressure. Unlike SWE-bench, it includes all the "long tail" of tasks (documentation, refactoring, debugging, planning) that benchmarks don't capture.
  • Mutation-testing rigour. Omega's gate runs cargo mutants on every non-trivial change. This gives unusually precise signal on whether test coverage is meaningful — harder to fake than coverage percentages alone.
  • Domain depth. The user has deep visibility into the codebase and can distinguish genuine improvements from benchmark over-fitting. This is the kind of domain expertise that academic evaluations rarely have.
  • The specific research angles below — §1.4 and §1.5 — which the literature has not studied in Omega's form.

§1.4 [a3f7c2] The human-orchestrated, artefact-returning sub-agent angle

The multi-agent literature (Kim et al. 2025, Anthropic's own analysis, Cemri et al. 2025) is almost entirely focused on LLM-orchestrated subagents: the parent LLM emits a tool call or REPL function invocation; the child runs; the child's output is a summary string that re-enters the parent's LLM context. This is the pattern those papers study, and it is the pattern they find wanting for most coding tasks.

But there is a structurally distinct third pattern that the literature does not appear to have studied rigorously: the human-orchestrated, artefact-returning sub-session.

In this pattern:

  • The orchestrator is the human, not the LLM. No orchestration tokens enter any model's context; no Baseline Paradox applies.
  • The child session runs to completion and returns artefacts on disk (commits, docs, test results), not a summary string in the parent's context. No context-pollution penalty.
  • The handoff is mediated by a seeded prompt that the LLM drafts and the human reviews before dispatching.

This is structurally closer to RLM (results land as artefacts, not strings in context) than to vanilla MAS, yet it avoids the failure modes that the literature documents for both.

Omega has been using this pattern throughout the planning project that produced this very document. The sub-sessions that wrote docs/sessionref-design-proposal.html, docs/rlm-exploration-proposal.html, and this file were all human-orchestrated, artefact-returning sub-sessions. The evidence base for this pattern is Omega's own commit history.

💡 [a3f7c2] Is the human-orchestrated, artefact-returning pattern a research angle worth formalising?

Nobody in the literature seems to have studied this regime rigorously. Omega is uniquely positioned to instrument and measure it. The key research questions are: does the pattern scale? does the handoff quality degrade as the child session scope grows? does the human-review step introduce meaningful error correction relative to fully-automated dispatch? These are empirically answerable with Omega's own session logs.

§1.5 [e91b54] The seeded-handoff working method, and the case for automating it first

The user has empirically observed — and this very planning project is the evidence — that the following pattern works very well in practice:

  1. A session runs until the LLM and the user converge on it being time to hand off (context drift, completion of a logical unit, or both).
  2. Handoff happens via a seeded prompt + model + effort, dispatched to a new or child session.
  3. The LLM drafts the seed; the user reviews and dispatches.

Two flavours share this mechanism:

  • Successor handoff: this session ends, a new one takes over linearly. The child continues the same work thread.
  • Sub-session handoff: this session continues, a child does a delimited task (write a doc, implement a feature, run a benchmark) and returns artefacts (commits, files). The parent later observes the results via git, the filesystem, or the event log.

Why this sidesteps the literature's critique of coding subagents

  • The orchestrator is human, not LLM — no orchestration-token overhead, no Baseline Paradox penalty.
  • The return is artefacts on disk, not summary strings in the parent's LLM context — no context-pollution penalty.
  • It is therefore structurally a third pattern, distinct from both REPL and vanilla subagents, and outside the scope of the critiques in §2.

The (semi-)automation opportunity

The earlier proposal framed handoff trigger detection ([549288]) as "manual first, automate later." This observation flips the priority: automating the working pattern we have already proved out may be a higher-EV investment than REPL or vanilla subagents.

  • It is incrementally automatable: trigger detection first (the agent flags when context drift suggests a handoff is due), then seed-prompt drafting (the agent writes the seed, human reviews), then dispatch (the agent initiates the child session). Each step gives daily value without the next.
  • It does not fight against the evidence base; it builds on a successfully demonstrated regime.
  • It is directly observable in Omega's own session logs — every seeded handoff is a data point, traceable via commit history.
⚠ [3c9a17] Should handoff automation (RQ-D) be the first investment, before REPL or subagents?

The seeded-handoff method is already producing results in daily use. REPL and vanilla subagents are not yet implemented and face real evidence headwinds (§2). The case for automating a known-good pattern before building unproven ones is strong. Not yet decided; see §6 and §8 for the full research question.

§2 Evidence base

Evidence is presented from most to least negative for multi-agent coding setups, with honest counter-evidence at the end. The agent survey (§2b) covers what existing coding agents actually ship.

Kim et al. (2025) — "Towards a Science of Scaling Agent Systems"

This paper evaluates every major multi-agent topology (parallel, sequential, centralised, hierarchical) across SWE-bench Verified and TerminalBench, with careful equal-budget controls. It is the most direct evidence on whether multi-agent architectures improve coding-agent performance.

⚠ Key findings — negative for MAS in coding
  • SWE-bench: −2% to −15% across all MAS topologies relative to the single-agent baseline at equal token budget. No topology consistently outperformed a well-tuned single agent.
  • TerminalBench Centralised: −19.2%. The centralised multi-agent topology (closest to vanilla subagent orchestration) shows the largest regression on the terminal-based coding benchmark.
  • Baseline Paradox. Stronger single-agent baselines make multi-agent improvements harder to demonstrate. As frontier models improve, the marginal value of adding orchestration complexity shrinks. This is a structural argument against investing in MAS before the single-agent baseline is mature.
  • Tool-Heavy Penalty. Coding agents that rely heavily on tool calls (file edit, shell execution) suffer disproportionately from multi-agent overhead. Each agent boundary adds latency and error propagation risk. Omega is tool-heavy by design.

Anthropic engineering post — multi-agent research system

Anthropic published a post-mortem on their internal multi-agent research system, written by engineers who built and operated it. The conclusions are notable coming from the team that builds the underlying models:

Multi-agent orchestration is the wrong approach for most coding tasks. The overhead of coordination, the error-propagation across agent boundaries, and the difficulty of debugging multi-agent trajectories outweigh the benefits for the majority of software-engineering work.

The post notes that the use cases where multi-agent does help are those with natural parallelism and clear artefact boundaries — not the general "do a coding task" shape that Omega primarily handles.

Citation: Alex L. Zhang, Tim Kraska, Omar Khattab. MIT OASYS Lab. arXiv:2512.24601.

Introduces RLM: treating the user prompt as a Python variable in a persistent REPL rather than raw context input. Benchmarks four tasks at extreme context lengths (6M–11M tokens for BrowseComp+; effectively ∞ for OOLONG-Pairs).

SRLM — Self-Reflective Program Search (Apple, March 2026)

Citation: Alizadeh, Shojaee, Cho, Farajtabar — Apple. arXiv:2603.15653.

⚠ Critical finding — recursion is not the primary driver
  • SRLM without sub-calls frequently beats full RLM. With GPT-5 on BrowseComp+: 94.6% vs. RLM-full 86%. The gains come from programmatic context interaction, not recursive model invocation.
  • RLM hurts on short contexts. §3.5 explicitly states: "When the context length falls within the model's native context window (<131K tokens), recursive RLM reasoning can degrade performance relative to the base model."
  • Omega's typical session is well under 131K tokens of code and shell output. This puts it squarely in the degradation regime.

Tran & Kiela (2025) — equal-budget single-agent parity

This paper asks whether the gains attributed to multi-agent systems survive equal-budget controls (i.e., giving the single agent the same total token budget as the multi-agent system). Across a range of coding and reasoning benchmarks, the finding is:

⚠ Equal-budget single agents match or beat MAS

Under equal token budgets, a single agent with a larger compute allocation matches or outperforms multi-agent systems on coding tasks. The gains reported for MAS in earlier work were largely an artefact of giving the multi-agent system more total tokens than the single-agent baseline. This is the equal-budget parity finding: orchestration adds complexity without adding performance when budgets are controlled.

Cemri et al. (2025) — 14 failure modes of multi-agent systems

A systematic empirical study of how multi-agent systems fail in practice, identifying 14 distinct failure modes. For coding agents specifically, the most impactful are:

  • Error propagation: an error in an early agent propagates silently through subsequent agents, compounding rather than being corrected.
  • Context loss at boundaries: the summary string returned by a child agent loses critical context that would have been visible to a single agent reading the full task.
  • Coordination overhead: the cost of specifying subtasks clearly enough for autonomous agent execution exceeds the cost of doing the task directly.
  • Debugging opacity: multi-agent trajectories are harder to inspect and correct than single-agent ones, because the parent agent cannot inspect the child's reasoning.
💡 Relevance to Omega's seeded-handoff pattern (§1.5)

Several of these failure modes are mitigated by human orchestration. The human reviewer catches error propagation before dispatch; the seed prompt preserves context rather than summarising it; the human can inspect and steer both sessions. The §1.5 pattern sidesteps 3 of the 4 failure modes above by design.

Counter-evidence — Open SWE and AdaptOrch

The evidence base is not uniformly negative. Two bodies of counter-evidence are worth representing honestly:

Open SWE-bench top submissions and AdaptOrch

SWE-bench top submissions (multi-agent)

As of mid-2026, several top-tier SWE-bench Verified submissions use multi-agent architectures — specifically specialised subagents for exploration, planning, and execution phases. These systems do outperform pure single-agent baselines on the benchmark. The caveat (consistent with Kim et al.) is that the gains materialise when subagents are highly specialised for distinct task phases rather than general-purpose clones of the parent. Phase-specialised orchestration may be a better model than capability-general orchestration.

AdaptOrch

AdaptOrch introduces adaptive orchestration: the system dynamically decides whether to use multi-agent or single-agent execution based on task complexity signals. On benchmarks where tasks vary widely in complexity, AdaptOrch outperforms both static single-agent and static multi-agent baselines. This is the most credible counter-case: not "always multi-agent" but "multi-agent when it actually helps, single-agent otherwise." This is consistent with RQ-D's thesis (automate the already-working handoff pattern rather than adding always-on orchestration).

CodeAct — Executable code actions (ICML 2024)

Citation: Xingyao Wang et al. ICML 2024. arXiv:2402.01030.

ℹ CodeAct findings and the important nuance
  • Up to 20% higher success rate on M3ToolEval vs. JSON/text action formats. 30% fewer turns.

  • Crucial nuance: "CodeAct" means code as the action format inside a ReAct loop, not a stateful REPL with accumulated variables. Omega could adopt code-as-action for tool calls without the full REPL machinery — this is a more conservative RQ-A that may deserve its own sub-question.

§2b Comparative agent survey [b2e97e]

Five coding agents were surveyed across REPL availability, subagent support, invocation primitive, and parent–child data flow.

opencode — ~/opencode/

  • REPL: No (shell access via Bash tool; no stateful interpreter namespace).
  • Subagents: Yes — via tool_use (task/agent tool ID).
  • Data flow: Call-return. Child produces final text; returned as tool result string in parent's LLM context. Sessions resumable via task_id.
  • Notable: Foreground and background (concurrent, auto-deny-prompts) modes. Result wrapped in XML tags.

forgecode — ~/forgecode/

  • REPL: No.
  • Subagents: Yes — agent names surface as tool names.
  • Data flow: Call-return. Final text returned as ToolOutput XML string in parent's LLM context. Can reuse existing ConversationId (stateful).
  • Notable: Interrupts propagate — parent abort triggers AgentToolInterrupted.

pi-mono — ~/pi-mono/

  • REPL: No (JavaScript renderer in UI is display-only, not agent-level).
  • Subagents: No — single-agent loop; no orchestration primitives.

fast-rlm — ~/fast-rlm/

  • REPL: Yes — Pyodide (WebAssembly Python) inside Deno. Sandboxed.
  • Subagents: Yes — via await llm_query(prompt, schema) in REPL namespace.
  • Data flow: Results land as Python variables (Pydantic-validated), not as strings in parent's LLM context. Parallel spawning via asyncio.gather.
  • Notable: Sub-agents do not inherit parent tools. Two runtimes (Python + Deno).

Claude Code — web docs

  • REPL: Partial — sandboxed code execution is an API-level beta tool, not user-facing.
  • Subagents: Yes — via Agent (formerly Task) tool_use. Anti-nested (subagents cannot spawn sub-subagents).
  • Data flow: Call-return. Result string in parent's LLM context. Background agents run concurrently (auto-deny-prompts). Per-subagent model and tool config.
  • Notable: Session forking as a separate primitive. Optional persistent memory directory per subagent.

§3 RQ-A REPL alone — value, design, research plan

The REPL-only cell (repl=on, subagents=off) — CodeAct-style persistent code execution without recursive model invocation.

The value case

A persistent REPL session accumulates state across tool calls: working directory, environment variables, loaded data, defined functions. For coding tasks this is potentially useful, but the CodeAct paper's key nuance applies: the advantage of code-as-action comes from action format, not necessarily from persistent state. Omega could adopt code-as-action for tool calls without the full REPL machinery.

The SRLM paper's warning about short contexts (<131K tokens) also applies here: if the REPL adds overhead (warmup, per-command round-trip, replay complexity on resume) without meaningful benefit at Omega's context sizes, it degrades rather than improves performance.

Design notes (if pursued)

  • Language [368041]: Shell (PTY-based) is the recommended starting point — zero new runtime dependencies, matches Omega's coding-agent persona, more deterministic for replay-based resume than Python. Python (via PyO3 or Pyodide) only if data-science workloads emerge.
  • Integration shape [fa3e44]: REPL as another tool alongside the existing tool set, not as a replacement. The LLM chooses the REPL when accumulated state is valuable; individual tools for one-shot operations.
  • Persistence [c88a22]: Event-replay (each executed command recorded as a ReplExecuted event; replayed on resume). More deterministic for shell than Python. Snapshot and hybrid strategies deferred.

Research plan for RQ-A

  1. Implement shell REPL behind repl=on flag.
  2. Run Harbor sweep: repl=off vs. repl=on (with subagents off in both), same task set, same token budget.
  3. Measure: gate passage rate, tokens/commit, time/commit, tool-call error rate. Threshold: at least neutral on gate passage, positive on at least one other metric.

§4 RQ-B Subagents alone — value, open question on viability

The subagents-only cell (repl=off, subagents=on) — vanilla subagent orchestration where the child session returns a summary string to the parent's LLM context.

⚠ [7d2e8f] Is RQ-B worth pursuing at all?

The evidence base (Kim et al., Cemri et al., Anthropic post, equal-budget parity finding) converges on a negative answer for coding tasks. Vanilla subagents specifically exhibit the failure modes the literature most consistently documents: context loss at boundaries (summary string loses detail), error propagation (parent cannot inspect child's reasoning), and coordination overhead.

The counter-evidence (specialised phase agents in SWE-bench, AdaptOrch) suggests that highly specialised subagents in a narrow task decomposition can help. Whether Omega's tasks decompose this way is an empirical question.

Not yet decided: whether to invest in RQ-B before empirical validation from RQ-D (which avoids the known failure modes).

Design notes (if pursued)

The Endeavor 1 types (SessionId, EventId, SessionRef, Origin::SubagentOf) already provide the primitives. The remaining design questions are:

  • Session model: in-process multi-session server (HashMap<SessionId, ActiveSession>) vs. separate Omega process per subagent. Leaning: separate process for crash isolation; revisit at design time.
  • Spawn tool JSON schema; SubagentSpawned / SubagentReturned event shapes (partially designed in Endeavor 1).
  • Phase 1 items 1.5 (UI surfacing) and 1.6 (WS migration) are prerequisites before any Phase 3 implementation.

§5 RQ-C RLM — REPL + subagents combined

The full RLM cell (repl=on, subagents=on) — the MIT paper's original design where llm_query is a function call inside the REPL, results land as Python variables, and recursion depth is bounded.

The case for RLM

For large-context retrieval and aggregation (6M–11M tokens), RLM's numbers are compelling: 91.3% on BrowseComp+ vs. 70.5% for summary agents, 58% on OOLONG-Pairs vs. 0.1% base. If Omega's workloads ever shift toward deep research tasks or very large codebase analysis, the domain gap narrows and RLM's advantages become relevant.

The case against RLM now

  • Domain gap: Omega's tasks are software-engineering at <131K tokens — exactly the regime where SRLM §3.5 shows RLM can hurt.
  • Depth ablation: for weaker models on CodeQA, more recursion means worse performance. Claude Sonnet is a strong model, but the trend is still concerning for coding tasks.
  • Two foreign runtimes (Pyodide-in-Deno for the reference implementation) add significant operational complexity.
  • RQ-A (REPL alone) must be validated first — RLM is only worth investigating if the REPL itself provides positive signal.

Dependency on RQ-A and RQ-B

RQ-C is blocked on positive signal from both RQ-A (REPL is beneficial in isolation) and at least neutral signal from RQ-B (subagents are not actively harmful). Building RLM before validating its components independently would make it impossible to attribute outcomes.

§6 RQ-D Handoff automation — the highest-EV candidate

Automate the seeded-handoff working method described in §1.5. Unlike RQ-A/B/C, this direction builds on a pattern that has already proved its value in daily use.

The value case

The seeded-handoff pattern (§1.5) is already producing results: it is how the planning documents for Omega were written, how feature implementation sub-sessions are dispatched, and how context drift is managed across sessions. The pattern works; the question is whether automating parts of it produces measurable improvement over the current manual flow.

The structural advantages over RQ-A/B/C:

  • Already validated. No hypothesis to test about whether the pattern helps — it demonstrably does. Automation is a quality-of-life and consistency improvement, not a gamble on an unproven approach.
  • Avoids the literature's failure modes. Human orchestrator, artefact returns — see §1.4.
  • Incrementally automatable. Each step gives daily value independently: trigger detection → seed-prompt drafting → dispatch.

Incremental automation steps

  1. Trigger detection. The agent monitors context growth, turn count, and topic coherence and flags when a handoff is likely beneficial. Surfaces a suggestion to the user without acting automatically. Low risk; high daily value.
  2. Seed-prompt drafting. When a handoff is flagged (or requested), the agent drafts the seed: a concise description of the current state, the next goal, and any necessary context for the incoming session. User reviews and edits before dispatch.
  3. Dispatch. The agent initiates the child session with the reviewed seed, correct model, and effort. The parent records the SessionRef to the child in its event stream, making the relationship navigable.

Research plan for RQ-D

  1. Instrument the current manual seeded-handoff workflow in Omega's event log: record each handoff event, the seed used, and the outcome (gate passage, commit quality, time elapsed).
  2. Establish baseline metrics over a sample of seeded-handoff sessions from everyday use.
  3. Implement trigger detection (Step 1). Measure: does automatic flagging agree with user-initiated handoffs? False-positive rate?
  4. Implement seed-prompt drafting (Step 2). Measure: time savings on seed writing; user edit rate; outcome quality of dispatched sessions.
  5. Compare outcomes (gate passage, tokens/commit, commit quality) between sessions that used automated seeding vs. manual seeding.

§7 What we'd benchmark

For RQ-A, RQ-B, RQ-C — Harbor flag-matrix sweeps

Metrics: gate passage rate, tokens/commit, time/commit, tool-call error rate, mutation-test survivor rate (where applicable).

Tasks: a fixed set covering: bug fix (with test), feature addition, refactor, documentation. Tasks must be representative of Omega's actual workload, not cherry-picked for multi-agent visibility.

For RQ-D — instrumented everyday use

RQ-D cannot be effectively evaluated by benchmark alone; the seeded-handoff pattern is inherently a multi-session, real-task phenomenon.

  • Primary data source: Omega's own session logs and commit history. Every seeded handoff leaves a trace (seed prompt, child session ID, outcome). Instrument the log reader to emit a structured handoff event when the pattern is detected.
  • Metrics: tokens consumed per commit (across parent + child sessions combined), time elapsed from task-start to gate-passing commit, gate-passage rate, mutation-test survivor rate in the child session's output.
  • Experimental substrate: the user's daily Omega work. The experimental condition is automation level (fully manual vs. trigger- detected vs. seed-drafted). Within-user comparison controls for task-difficulty variance.

§8 Open questions

Ground rules (already decided)

✓ [b0bc06] Feature flags — REPL and subagents independently switchable

Two independent runtime flags (OMEGA_FEATURE_REPL=1, OMEGA_FEATURE_SUBAGENTS=1), both defaulting to off, recorded in SessionStartedEvent.features. Harbor sweeps combinations without rebuild.

✓ [abe7e7] Stay on develop; feature flags alone, no branch

All new code lands on develop behind flags. The gate runs every commit. No long-lived feature branch.

RQ-A questions

💡 [368041] What language is the REPL?

Lean: shell (PTY-based). Zero new runtime dependencies; deterministic enough for replay-based resume; matches Omega's coding-agent persona. Python (via PyO3 or Pyodide-in-Deno) only if data-science workloads emerge. Rust-native scripting (rhai/rune) rejected — LLM proficiency too low, Python ecosystem lost.

💡 [fa3e44] How do existing tools integrate with the REPL?

Lean: REPL as another tool, not a replacement. Existing tool set remains. A shell_session tool is added that feeds commands into the persistent session. The LLM chooses REPL when state accumulation is valuable. All three production agents surveyed use this approach.

💡 [c88a22] How does REPL state persist across process restarts?

Lean: event-replay. Each executed command is a ReplExecuted event. On resume, commands are replayed in order. Shell commands are sufficiently deterministic for this to work. Snapshot (binary blobs in events.jsonl) violates Contract Authority. Hybrid deferred.

💡 [835f01] Cost / determinism / debuggability

Real costs: PTY startup latency, per-command round-trip, replay non-determinism, harder trajectory debugging. For shell REPL: startup is fast; determinism is sufficient; ReplExecuted event log makes trajectories inspectable. For RLM-style llm_query recursion: additional LLM call budget analogues of max_depth and max_money_spent (from RLMConfig) will be needed.

💡 [c86aa6] Interaction with strict resume

When REPL is built, the round-trip gate test must be extended to verify REPL state equivalence after resume. For shell REPL, "equivalent" means working directory and key environment variables match (not byte-for-byte stdout identity). This extension is a prerequisite for the REPL iteration, not for vanilla-subagents or handoff-automation work.

RQ-B questions

⚠ [7d2e8f] Is RQ-B even worth pursuing?

Kim et al., Cemri et al., the Anthropic engineering post, and the equal-budget parity finding (Tran & Kiela) all suggest caution. The counter-evidence (specialised phase agents, AdaptOrch) requires a careful task-decomposition design that may not generalise. This question should be revisited after RQ-D's seeded-handoff instrumentation provides real data on what Omega's actual subtask decomposition looks like in practice.

RQ-C questions

💡 [cf2db0] REPL first or subagents first?

Lean: subagents first (if RQ-B is pursued at all). Research + survey consensus supports subagents without REPL for coding tasks. REPL persistence is unsolved and adds two foreign runtimes if Python is chosen. Subagents re-use existing Omega infrastructure and are stepping stones toward RLM, not away from it. Not yet decided.

💡 [8e161a] How does an llm_query call relate to SessionId?

Lean: each llm_query invocation is its own session — own SessionId, own events.jsonl, own folder. Mirrors the existing decision that subagents are child sessions. Only the invocation surface changes (REPL function call vs. tool_use); the underlying spawn-a-child-session machinery is shared.

💡 [053c4a] What is "the session" in an RLM Omega?

Lean: today's framing survives intact. "The session" is the outer Omega instance the user interacts with. Its REPL is a property of that instance. Children spawned by llm_query are child sessions in exactly the same sense as vanilla subagents.

💡 [bab784] User interaction with an RLM Omega

Lean: chat messages stay; REPL is internal; UI optionally exposes REPL state as collapsible ReplExecuted event blocks. A dedicated "REPL console" pane is deferred.

💡 [1f28f7] Output format — autoregressive vs. variable

Lean: autoregressive prose at the user boundary; variable returns only within the REPL and in subagent-to-parent data flow. FINAL(variable)-style returns may apply within subagent calls but not to the user-facing reply.

RQ-D questions

⚠ [3c9a17] Should handoff automation (RQ-D) be the first investment, before REPL or subagents?

The seeded-handoff pattern is already validated in daily use. REPL and vanilla subagents are not yet implemented and face real evidence headwinds. The case for automating a known-good pattern before building unproven ones is strong. The incrementally-automatable nature of RQ-D means the first step (trigger detection) can deliver value without committing to the full design. Not yet decided.

💡 [e91b54] Seeded-handoff automation — how far to go first?

Three incremental steps (§6): trigger detection, seed-prompt drafting, dispatch. Each step is independently useful. The research question is: how much of the handoff does the agent need to automate before human review becomes the bottleneck rather than the safeguard? Empirical question; answer from instrumented use.

💡 [a3f7c2] Is the human-orchestrated, artefact-returning pattern worth formalising as a research object?

The pattern is distinct from both vanilla MAS and REPL-based RLM. It sidesteps the known failure modes. The literature has not studied it rigorously. Omega's session logs and commit history constitute a growing evidence base. The concrete research design is: instrument the handoff events, measure outcomes, compare against sessions that did not use handoffs. Feasibility: high. Unique Omega advantage: high.
