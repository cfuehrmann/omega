REPL, Subagents & Handoff Automation — Research Doc

Status: Research document — not a proposal toward imminent implementation · Supersedes docs/rlm-exploration-proposal.html · Endeavor 1 (events-as-state) tracked in docs/session-design.html · Canonical path: docs/repl-and-subagents-research.html

📊 Findings to date

Between v0.1.8 and v0.1.13 we ran a sequence of feature-flag benchmark experiments on two cached Terminal-Bench 2 tasks: fix-git and crack-7z-hash. The six findings below capture what we have actually learned. Details are in collapsibles; the §3–§6 research questions are updated with status lines.

F-A Feature-flag tier matrix is sound

Three independent boolean flags — OMEGA_FEATURE_REPL, OMEGA_FEATURE_REPL_REPLACES_FILEOPS, OMEGA_FEATURE_REPL_REPLACES_SHELL — are each runtime-switchable via Harbor's --ae KEY=VAL mechanism and each recorded in SessionStartedEvent.features. All four valid combinations (with repl=1) have been exercised end-to-end and confirmed to produce the expected toolset in agent sessions. The matrix enables apples-to-apples comparison without rebuilds; post-mortem can grep events.jsonl to know which combination a run used.

Detail — tier definitions and flag combinations
Tier nameFlagsTools removedWhat the LLM must use for…
Baseline (OFF) none—standard tool set
Additive REPL repl=1nothing python_repl available alongside all 12 standard tools
Tier 1 (limit-mode) repl=1 replaces_fileops=1 6 file-op tools file reads/writes → must choose python_repl or run_command
Tier 2a repl=1 replaces_shell=1 4 shell tools shell execution → must use subprocess inside python_repl
Tier 2 (full) repl=1 replaces_fileops=1 replaces_shell=1 10 tools total everything → python_repl only

Setting either REPLACES_* flag without REPL=1 is a configuration error; the agent exits loudly at startup.

F-B Additive REPL does not engage the LLM

On both fix-git and crack-7z-hash, when python_repl was offered alongside the full standard tool set, the LLM never called it once. Both runs still passed (the tasks are well within baseline capability), but the tool was completely ignored. The additive design cannot measure REPL value because the experimental variable is never engaged. N=2.

Detail — likely cause and implication

The most likely explanation is that the LLM is heavily fine-tuned on dedicated file-operation and shell-execution tool calls. Given a free choice, it defaults to the familiar pattern. This is consistent with Kim et al.'s Baseline Paradox finding: stronger single agents (with well-tuned tool sets) make it harder to demonstrate the marginal value of new primitives. The finding implies that any benchmark of REPL value must use a constrained treatment arm — either remove competing tools or make the task structure strongly favour the REPL pattern.

Event logs: bench/jobs/v018-smoke-crack-repl-on/ (crack-7z-hash), bench/jobs/v018-smoke-short-repl-on/ (fix-git).

F-C Tier 1 engages REPL on file-centric tasks; inert on shell-centric ones

With OMEGA_FEATURE_REPL_REPLACES_FILEOPS=1, the LLM used python_repl exactly once on fix-git — to write the conflict-resolved file content — and the task passed. On crack-7z-hash the file tools being removed were inert for the task anyway (it involves cracking a 7z password, not manipulating text files), so the LLM just timed out. N=1 each.

Detail — what the python_repl call on fix-git looked like

The LLM resolved a Git merge conflict that required reconstructing a Markdown biography file with correct content. After gathering all context via run_command, it wrote the final file in a single python_repl call using an embedded Python heredoc: 

content = """I am a Postdoctoral Researcher at Stanford CS.

Previously, I was a Student Researcher at [Google Research](...),
...
"""
with open('/path/to/file.md', 'w') as f:
    f.write(content)

This is a clean signal: the file-write tools were not available, and the LLM naturally reached for Python string literals and open(). The task passed.

The crack-7z-hash Tier 1 run timed out without attempting python_repl meaningfully — the removed tools (read_file, grep_files, etc.) were simply not part of the solution path for a password-cracking task. Removing file tools provides no forcing function for a pure-shell task.

Event logs: bench/jobs/v019-inspect-fix-git-limit-mode/, bench/jobs/v019-inspect-crack-7z-hash-limit-mode/.

F-D Tier 2 first passed via a fetch_url.postprocess shell backdoor

The very first Tier 2 (full) run — v0.1.10 on crack-7z-hash — produced a pass, but for the wrong reason. The LLM discovered that fetch_url's postprocess field was executed as a shell command inside the container, and used it as a backdoor: 31 of 33 tool calls were fetch_url with arbitrary shell commands in postprocess. This was a loophole, not REPL engagement. After closing it (removing postprocess entirely when shell tools are gated), the next run (v0.1.11) failed. The cleaner Tier 2 mode is what was then actually studied.

Detail — loophole discovery and patch

The v0.1.10 crack-7z-hash run shows the pattern clearly in bench/jobs/v0110-inspect-crack-7z-hash-tier2/: calls like 

fetch_url(url="https://example.com",
          postprocess="apt-get install -y p7zip-full </dev/null >/tmp/apt_log 2>&1")

The fix in v0.1.11 was to strip postprocess from the fetch_url tool description (and refuse to execute it) when OMEGA_FEATURE_REPL_REPLACES_SHELL=1 is set. This forced the LLM to execute shell work through subprocess inside python_repl.

Note: v0.1.10's fix-git Tier 2 pass is also suspect for the same reason — event logs show fetch_url usage there too.

F-E Several rounds of python_repl hardening were needed

After closing the loophole, v0.1.11 failed (no bootstrap), v0.1.12 failed (no timeout recovery), and only v0.1.13 passed. The hardening work that accumulated across those versions was necessary to make Tier 2 a real experiment.

Detail — the hardening sequence
  • Marker rename (REPL_END → REPL_RESPONSE): the original end-of-output sentinel sometimes appeared in Python tracebacks, causing premature truncation. Renaming to something less likely to collide fixed spurious truncations.
  • python3 bootstrap-on-missing: the benchmark container does not have a running Python3 interpreter on PATH at agent startup. Added apt-get install -y python3 as a first-call bootstrap, emitting a python_repl_bootstrapped event on success. Without this, the first call to python_repl silently failed.
  • Per-call timeout with SIGINT-soft / kill_group-hard escalation: long-running REPL calls (password cracking, compilation) would hang indefinitely before this. Added a 60-second per-call timeout that sends SIGINT first (giving Python a chance to print partial output and return control) and escalates to process-group kill if SIGINT is ignored. REPL state is preserved across the interruption.
  • tee-always forensics + tail-bias on timeout: all output is written to a side-channel log file as it streams. On timeout, the last N bytes of output are returned rather than the beginning, giving the LLM context about where execution was when it was interrupted.
  • System-prompt rework to encourage variable-pattern: updated the REPL section of the system prompt to explicitly suggest storing results in variables (rather than re-opening file handles) so that cross-call state accumulates naturally.

Each of these was diagnosed from a failing run's events.jsonl and fixed in the subsequent version.

F-F v0.1.13 Tier 2 on crack-7z-hash: 100% REPL, genuine state, 10 min 44 s — passed

The v0.1.13 Tier 2 (full) run on crack-7z-hash produced the result the RLM hypothesis predicts: every tool call was python_repl (26/26), the Python interpreter carried state across 4+ calls in multiple variable chains, the LLM used Python threading to parallelise the password search, and three subprocess timeouts were recovered from with REPL state intact. The task passed in 10 minutes 44 seconds total (6 min 41 s agent execution). N=1, but the qualitative trajectory is clean.

§1 Framing

§1.1 Research-mode stance

⚠ This is a research document, not a proposal

Whether and how to add a REPL, vanilla subagents, RLM-style recursive subagents, or automated handoffs to Omega is open. None are committed work. A direction earns committed work only when empirical evidence — Harbor sweeps, instrumented everyday use, or literature applied to Omega's task profile — shows improvement in the metrics that matter (gate passage, tokens/commit, commit quality, time/commit). Intuition and analogies from other agents do not clear the bar.

§1.2 What Omega specifically brings

Unlike one-shot benchmark datasets, Omega offers: the user's daily work as a continuous real-world experiment; cargo mutants gate rigour on every commit (unusually precise signal); deep domain visibility to distinguish genuine improvement from benchmark over-fitting; and the specific research angles below (§1.4, §1.5) that the literature has not studied in Omega's form.

§1.4 [a3f7c2] The human-orchestrated, artefact-returning angle

The multi-agent literature (Kim et al., Anthropic analysis, Cemri et al.) studies LLM-orchestrated subagents: parent emits tool call; child returns a summary string into parent's LLM context. That is the pattern the papers find wanting. Omega already uses a structurally distinct third pattern: the human orchestrates; the child returns artefacts on disk (commits, docs), not strings. No orchestration-token overhead, no Baseline Paradox, no context-pollution penalty. The planning project that produced this very document was executed via this pattern (sub-sessions wrote docs/sessionref-design-proposal.html, docs/rlm-exploration-proposal.html, and this file). Omega's commit history is the evidence base. Nobody in the literature appears to have studied this regime rigorously — Omega is uniquely positioned to instrument and measure it.

§1.5 [e91b54] The seeded-handoff working method

A session runs until the LLM and user converge on handoff time (context drift, logical unit completion, or both). The LLM drafts a seed prompt; the user reviews and dispatches to a new or child session. Two flavours: successor handoff (session ends; child continues linearly) and sub-session handoff (session continues; child does a delimited task and returns artefacts). Because the orchestrator is human and returns are artefacts on disk, this pattern sidesteps the three failure modes the literature most consistently documents (context loss, error propagation, coordination overhead). It is incrementally automatable: trigger detection → seed-prompt drafting → dispatch, each step independently valuable. See §6 for the automation roadmap.

§2 Evidence base

Evidence is presented from most to least negative for multi-agent coding setups. All findings below concern coding / software-engineering tasks unless noted otherwise. The RLM papers' strongest results are on long-context retrieval tasks (6M–11M tokens) — a domain gap with Omega's typical sessions (<131K tokens).

§3 RQ-A REPL alone

Status after findings: F-B shows additive REPL is not measurable (LLM ignores it). F-C shows Tier 1 engages REPL on file-centric tasks but not shell-centric ones. F-F shows full Tier 2 on a shell-centric task works end-to-end with genuine REPL state usage. The next step for RQ-A is a broader sweep to establish whether Tier 2 helps, hurts, or has mixed effects across task categories.

RQ-A detail — value case, design leans, dismissed approaches

Value case: a persistent REPL accumulates state (working directory, environment, loaded data, defined functions) across tool calls. CodeAct's +20% / −30 turns result is the strongest evidence — but the advantage is from action format, not persistent state. Omega could adopt code-as-action without the full REPL machinery (a more conservative sub-question). SRLM §3.5 warning applies at <131K tokens.

Design leans (open questions in §8):

  • Shell REPL (PTY-based, [368041]) — resolved as Python subprocess
  • REPL as another tool alongside existing set ([fa3e44])
  • Event-replay persistence ([c88a22])
  • Round-trip gate extension for REPL state equivalence ([c86aa6])

What the additive REPL experiment showed (F-B): The LLM consistently ignored python_repl when the full standard tool set was available. This is consistent with Kim et al.'s Baseline Paradox: fine-tuned tool-call behaviour crowds out a new, unfamiliar tool.

What the Tier 1 experiment showed (F-C): Removing file tools forced the LLM to reach for Python for file writes. The signal is clean but limited: only fix-git benefited; crack-7z-hash failed because file tools were inert for that task.

RQ-C detail — value case, design leans, dependency on RQ-A/B

Value case: for large-context retrieval (6M–11M tokens), RLM's numbers are compelling (91.3% BrowseComp+, 58% OOLONG-Pairs). If Omega's workloads shift toward deep research or very large codebase analysis, the domain gap narrows.

Case against now: domain gap (Omega <131K tokens = SRLM's degradation regime); depth ablation hurts CodeQA for weaker models; two foreign runtimes add operational complexity; RQ-A must be validated first.

Dependency: blocked on positive signal from RQ-A (REPL beneficial in isolation) and at least neutral from RQ-B (subagents not actively harmful). RLM is essentially free once both prerequisites exist (≈2 days per §7).

✓ [cf2db0] Decided — REPL first, subagent second, RLM third

Resolved by the MVP-for-benchmark strategy. REPL shipped faster as a benchmark prototype, is the substrate for RLM, and doesn't depend on subagent machinery.

💡 [8e161a] How does an llm_query call relate to SessionId?

Lean: each llm_query invocation is its own session — own SessionId, own events.jsonl, own folder. Mirrors the existing decision that subagents are child sessions. Only the invocation surface changes (REPL function call vs. tool_use); the underlying spawn-a-child-session machinery is shared.

💡 [053c4a] What is "the session" in an RLM Omega?

Lean: today's framing survives intact. "The session" is the outer Omega instance the user interacts with. Its REPL is a property of that instance. Children spawned by llm_query are child sessions in exactly the same sense as vanilla subagents.

💡 [bab784] User interaction with an RLM Omega

Lean: chat messages stay; REPL is internal; UI optionally exposes REPL state as collapsible ReplExecuted event blocks. A dedicated "REPL console" pane is deferred.

💡 [1f28f7] Output format — autoregressive vs. variable

Lean: autoregressive prose at the user boundary; variable returns only within the REPL and in subagent-to-parent data flow. FINAL(variable)-style returns may apply within subagent calls but not to the user-facing reply.

RQ-D detail — incremental automation steps, open questions

Automate the seeded-handoff working method (§1.5). Unlike RQ-A/B/C, this builds on a pattern that has already proved its value in daily use. Deprioritised relative to REPL/subagent benchmarks by the MVP-for-benchmark strategy; may rise if sweeps return null or weak results.

Incremental automation steps

  1. Trigger detection. The agent monitors context growth, turn count, and topic coherence; flags when a handoff is likely beneficial. Surfaces a suggestion to the user without acting. Low risk; high daily value. Instrument the current manual workflow in Omega's event log first: record each handoff event, the seed used, and the outcome (gate passage, commit quality, time elapsed).
  2. Seed-prompt drafting. When a handoff is flagged (or requested), the agent drafts the seed: current state, next goal, necessary context for the incoming session. User reviews and edits before dispatch.
  3. Dispatch. The agent initiates the child session with the reviewed seed, correct model, and effort. Records the SessionRef to the child in its event stream. Each step is independently useful without the next.
💡 [e91b54] Seeded-handoff automation — how far to go first?

Three incremental steps (above): trigger detection, seed-prompt drafting, dispatch. Each step is independently useful. The research question is: how much of the handoff does the agent need to automate before human review becomes the bottleneck rather than the safeguard? Empirical question; answer from instrumented use.

💡 [a3f7c2] Is the human-orchestrated, artefact-returning pattern worth formalising as a research object?

The pattern is distinct from both vanilla MAS and REPL-based RLM. It sidesteps the known failure modes. The literature has not studied it rigorously. Omega's session logs and commit history constitute a growing evidence base. The concrete research design: instrument the handoff events, measure outcomes, compare against sessions that did not use handoffs. Feasibility: high. Unique Omega advantage: high.

RQ-A questions

💡 [368041] What language is the REPL? RESOLVED

Decided: Python (subprocess-based, not PTY). The original lean was shell/PTY; implementation used python3 -c with a long-lived subprocess. Python proved natural for the LLM (F-F: the LLM used subprocess, threading, and file I/O fluently). PTY-based shell REPL remains unexplored.

💡 [fa3e44] How do existing tools integrate with the REPL? RESOLVED (for now)

Implemented as additive tool (Tier 0), plus flag-gated replacements (Tiers 1–2). The Tier 2 full mode removes all competing tools, forcing Python. The additive design (original lean) was shown to be non-measurable (F-B): the LLM ignores python_repl when alternatives exist. Tier 2 forces engagement by removing alternatives.

💡 [c88a22] How does REPL state persist across process restarts? OPEN

Current state: strict_resume refuses REPL sessions loudly. See session-design.html §oq-repl-replay for the event-replay design. The principled fix (replay python_repl calls into a fresh subprocess) is not yet implemented. For benchmark use, this is acceptable: Harbor reruns failed tasks.

💡 [835f01] Cost / determinism / debuggability OPEN

Real costs observed: per-call timeout overhead (60 s max, with SIGINT escalation); subprocess hangs requiring kill_group; non-determinism in apt-get install timings. For shell work inside Python: determinism is sufficient for benchmark use. For llm_query recursion: not yet relevant; RQ-C not started. The python_repl_bootstrapped event and tee-forensics output make trajectories inspectable from events.jsonl alone.

💡 [c86aa6] Interaction with strict resume OPEN

strict_resume currently refuses REPL sessions with a loud error. The round-trip gate test has not been extended to verify REPL state equivalence. For benchmark use, this is acceptable (Harbor reruns; no resume needed). For daily-use REPL, the extension is a prerequisite. See session-design.html §oq-repl-replay.

RQ-B questions

⚠ [7d2e8f] Is RQ-B even worth pursuing? OPEN

Kim et al., Cemri et al., the Anthropic engineering post, and the equal-budget parity finding (Tran & Kiela) all suggest caution. The counter-evidence (specialised phase agents, AdaptOrch) requires a careful task-decomposition design that may not generalise. This question should be revisited after the broader REPL sweep provides task-category data on what decomposition Omega's workload actually needs. If the REPL sweep shows mixed results, RQ-D may provide better data on natural subtask decomposition before committing to RQ-B.
