§1 Framing

§1.1 Research-mode stance

⚠ This is a research document, not a proposal

Whether and how to add a REPL, vanilla subagents, RLM-style recursive subagents, or automated handoffs to Omega is open. None are committed work. A direction earns committed work only when empirical evidence — Harbor sweeps, instrumented everyday use, or literature applied to Omega's task profile — shows improvement in the metrics that matter (gate passage, tokens/commit, commit quality, time/commit). Intuition and analogies from other agents do not clear the bar.

✓ [b0bc06] Ground rule — feature flags, no branch

Two independent runtime flags (OMEGA_FEATURE_REPL=1, OMEGA_FEATURE_SUBAGENTS=1), both defaulting to off, recorded in SessionStartedEvent.features. Harbor sweeps all four cells without rebuilds. All new code lands on develop behind flags — no long-lived feature branch. [abe7e7]

subagents=off subagents=on
repl=off baseline (current Omega) vanilla subagents
repl=on REPL only (CodeAct-style) RLM (REPL + llm_query)

RQ-D (handoff automation) falls outside this matrix; see §1.5 and §6.

§1.2 What Omega specifically brings

Unlike one-shot benchmark datasets, Omega offers: the user's daily work as a continuous real-world experiment; cargo mutants gate rigour on every commit (unusually precise signal); deep domain visibility to distinguish genuine improvement from benchmark over-fitting; and the specific research angles below (§1.4, §1.5) that the literature has not studied in Omega's form.

§1.4 [a3f7c2] The human-orchestrated, artefact-returning angle

The multi-agent literature (Kim et al., Anthropic analysis, Cemri et al.) studies LLM-orchestrated subagents: parent emits tool call; child returns a summary string into parent's LLM context. That is the pattern the papers find wanting. Omega already uses a structurally distinct third pattern: the human orchestrates; the child returns artefacts on disk (commits, docs), not strings. No orchestration-token overhead, no Baseline Paradox, no context-pollution penalty. The planning project that produced this very document was executed via this pattern (sub-sessions wrote docs/sessionref-design-proposal.html, docs/rlm-exploration-proposal.html, and this file). Omega's commit history is the evidence base. Nobody in the literature appears to have studied this regime rigorously — Omega is uniquely positioned to instrument and measure it.

§1.5 [e91b54] The seeded-handoff working method

A session runs until the LLM and user converge on handoff time (context drift, logical unit completion, or both). The LLM drafts a seed prompt; the user reviews and dispatches to a new or child session. Two flavours: successor handoff (session ends; child continues linearly) and sub-session handoff (session continues; child does a delimited task and returns artefacts). Because the orchestrator is human and returns are artefacts on disk, this pattern sidesteps the three failure modes the literature most consistently documents (context loss, error propagation, coordination overhead). It is incrementally automatable: trigger detection → seed-prompt drafting → dispatch, each step independently valuable. See §6 for the automation roadmap and §1.6 for why automation is sequenced after first benchmark data.

✓ [3c9a17] Decided — benchmark data first; handoff automation sequenced after

Originally posed as "should handoff automation be the first investment, before REPL or subagents?" Answered by §1.6: no. The MVP-for-benchmark strategy makes REPL/subagent/RLM data cheap to obtain; benchmarks then inform what workflow is worth automating. Handoff automation (RQ-D, §6) remains valuable and may rise in priority if benchmark sweeps return null or weak results.

§1.6 [f8a1e0] Path to first benchmark — build to measure, not to ship

The strategic question for RQ-A, RQ-B, and RQ-C is not "how do we build the polished feature" but "how do we get reliable benchmark data fastest?" The bar is: Harbor can sweep a feature flag and we can score the runs. Everything beyond that is deferred until after data exists.

What MVP allows us to cut

The benchmark MVP only needs enough machinery for Harbor to execute a task and produce a pass/fail signal. That drops a lot of normally-required production concerns:

Production concernNeeded for benchmark MVP?
Persistence / resumabilityNo — if the benchmark process dies, Harbor reruns the task
Event-sourcing every internal stateNo — useful for forensics but not for pass/fail
Crash isolation, OOM handlingNo — gross failures are themselves data points
SandboxingNo — Harbor runs in containers; threat model is "our code", not adversarial
UI integrationNo — Harbor reads programmatic results
Streaming progressNo — block until done
Nested recursionNo — depth=1 is enough for the headline benchmarks
Cancellation / kill semanticsNo — Harbor's wall clock provides natural timeout

What remains is the core experimental variable: does adding this primitive change the outcome distribution, holding everything else fixed?

Deliberate cut — preserves the existing tool-call loop, so the benchmark answers "does adding a Python REPL help?" independently of "should we rethink the orchestration model entirely?" Two questions, two experiments.

Subagent MVP sketch

A feature-flagged spawn_subagent tool that creates an in-process child ActiveSession, runs it to its first TurnEnd or TurnInterrupted, and returns the last assistant message as the tool result.

This is genuinely more work than REPL because:

  • Async coordination (parent suspended while child runs) needs care.
  • Event shapes are designed but have to be wired through.
  • Spawning from inside an LLM call is non-trivial.

Silver lining: SessionId, SessionRef, Origin, LoggedEvent, and strict resume already exist. Not starting from zero on subagents — starting from "Phase 1 + 2 done, Phase 3 to scaffold."

RLM MVP — essentially free given the prerequisites

Cheap once REPL and subagent MVPs both exist. The REPL exposes llm_query(task, model, effort) -> str as a Python function that calls into the subagent machinery. Result marshalling is just string-in / string-out.

Estimated: ~2 days, contingent on REPL and subagent MVPs.

✓ Recommended order: REPL → Subagent → RLM
  1. REPL is faster to ship in benchmarkable form. Earlier data.
  2. REPL is the substrate for RLM. Once REPL exists, RLM is cheap to add.
  3. Subagent alone is the case the evidence is most skeptical of (Kim et al., Anthropic engineering post). Putting it second means we can skip it if REPL + RLM data is already conclusive.
  4. REPL MVP doesn't depend on subagent, so we can ship and measure cleanly without coordinating two new systems at once.

Total: roughly 2–3 weeks elapsed for the full sweep, depending on how sequentially the work runs. Each stage produces benchmark data independently.

⚠ Harbor sweep mechanics

Harbor sweeps the four-cell flag matrix against a fixed task set covering: bug fix (with test), feature addition, refactor, documentation. Tasks must be representative of Omega's actual workload, not cherry-picked for multi-agent visibility. Metrics: gate passage rate, tokens/commit, time/commit, tool-call error rate, mutation-test survivor rate. RQ-D uses instrumented everyday use rather than a Harbor sweep; see §6.

⚠ What this means for RQ-D (handoff automation)

Survives, but bumped down. The original framing of RQ-D as "highest-EV candidate" was premised on the assumption that benchmarking the unproven primitives would be costly. The MVP-for-benchmark strategy makes that cost much lower than the elegant-handoff investment would be. Net: benchmark data arrives faster than handoff automation could ever be polished. RQ-D remains documented (§6) and may rise in priority if benchmark sweeps return null or weak results.

§2 Evidence base

Evidence is presented from most to least negative for multi-agent coding setups. All findings below concern coding / software-engineering tasks unless noted otherwise. The RLM papers' strongest results are on long-context retrieval tasks (6M–11M tokens) — a domain gap with Omega's typical sessions (<131K tokens).

§3 RQ-A REPL alone

Value case: a persistent REPL accumulates state (working directory, environment, loaded data, defined functions) across tool calls. CodeAct's +20% / −30 turns result is the strongest evidence — but the advantage is from action format, not persistent state. Omega could adopt code-as-action without the full REPL machinery (a more conservative sub-question). SRLM §3.5 warning applies at <131K tokens.

Design leans (open questions in §8): shell REPL (PTY-based, [368041]); REPL as another tool alongside existing set ([fa3e44]); event-replay persistence ([c88a22]); round-trip gate must be extended for REPL state equivalence ([c86aa6]).

Next step: implement shell REPL behind repl=on flag; run Harbor sweep; measure gate passage rate, tokens/commit, time/commit, tool-call error rate vs. baseline. See §1.6 for the REPL MVP sketch and effort estimate.

⚠️ Empirical finding — v0.1.8 smoke (fix-git, crack-7z-hash)

Both tasks were run with OMEGA_FEATURE_REPL=1 (additive design: python_repl available alongside all twelve standard tools). In both sessions the LLM did not call python_repl once — it consistently chose run_command and the dedicated file tools (read_file, grep_files, etc.). Session events are in bench/jobs/v018-smoke-*/.

Implication for RQ-A: the additive REPL design is not measurable with the current LLM behaviour. The tool is present but ignored. To obtain benchmark data we need to create a context where the LLM must choose between python_repl and run_command — hence the OMEGA_FEATURE_REPL_REPLACES_FILEOPS limit-mode flag added in the next sub-session (see Next Steps above).

§4 RQ-B Subagents alone

⚠ [7d2e8f] Is RQ-B worth pursuing at all?

Kim et al., Cemri et al., the Anthropic post, and Tran & Kiela all converge negative for coding tasks. Counter-evidence (specialised phase agents, AdaptOrch) requires narrow task decomposition that may not generalise to Omega's workload. Not yet decided: whether to invest in RQ-B before RQ-D's seeded-handoff instrumentation provides real data on Omega's actual subtask decomposition.

Design leans: the Endeavor 1 types (SessionId, SessionRef, Origin::SubagentOf) already provide the primitives. Remaining: session model (in-process vs. separate process), spawn tool JSON schema, SubagentSpawned/SubagentReturned event shapes. Phase 1.5 (UI surfacing) and 1.6 (WS migration) are prerequisites before any Phase 3 implementation. See §1.6 for the Subagent MVP sketch and effort estimate.

§5 RQ-C RLM — REPL + subagents combined

Value case: for large-context retrieval (6M–11M tokens), RLM's numbers are compelling (91.3% BrowseComp+, 58% OOLONG-Pairs). If Omega's workloads shift toward deep research or very large codebase analysis, the domain gap narrows.

Case against now: domain gap (Omega <131K tokens = SRLM's degradation regime); depth ablation hurts CodeQA for weaker models; two foreign runtimes add operational complexity; RQ-A must be validated first.

Dependency: blocked on positive signal from RQ-A (REPL beneficial in isolation) and at least neutral from RQ-B (subagents not actively harmful). RLM is essentially free once both prerequisites exist (≈2 days per §1.6).

✓ [cf2db0] Decided — REPL first, subagent second, RLM third

Resolved by the MVP-for-benchmark strategy (§1.6). REPL ships faster as a benchmark prototype (≈4–5 days vs. ≈6–7 for subagent), is the substrate for RLM, and doesn't depend on subagent machinery. Subagent second because (a) most skeptically received by the evidence, so skip if REPL+RLM data is conclusive, and (b) async parent-child coordination is harder.

§6 RQ-D Handoff automation — sequenced after first benchmark data

Automate the seeded-handoff working method (§1.5). Unlike RQ-A/B/C, this builds on a pattern that has already proved its value in daily use. Deprioritised relative to REPL/subagent benchmarks by the MVP-for-benchmark strategy; may rise if sweeps return null or weak results.

Incremental automation steps

  1. Trigger detection. The agent monitors context growth, turn count, and topic coherence; flags when a handoff is likely beneficial. Surfaces a suggestion to the user without acting. Low risk; high daily value. Instrument the current manual workflow in Omega's event log first: record each handoff event, the seed used, and the outcome (gate passage, commit quality, time elapsed).
  2. Seed-prompt drafting. When a handoff is flagged (or requested), the agent drafts the seed: current state, next goal, necessary context for the incoming session. User reviews and edits before dispatch.
  3. Dispatch. The agent initiates the child session with the reviewed seed, correct model, and effort. Records the SessionRef to the child in its event stream. Each step is independently useful without the next.

Ground rules (already decided)

RQ-A questions

💡 [368041] What language is the REPL?

Lean: shell (PTY-based). Zero new runtime dependencies; deterministic enough for replay-based resume; matches Omega's coding-agent persona. Python (via PyO3 or Pyodide-in-Deno) only if data-science workloads emerge. Rust-native scripting (rhai/rune) rejected — LLM proficiency too low, Python ecosystem lost.

💡 [fa3e44] How do existing tools integrate with the REPL?

Lean: REPL as another tool, not a replacement. Existing tool set remains. A shell_session tool is added that feeds commands into the persistent session. The LLM chooses REPL when state accumulation is valuable. All three production agents surveyed use this approach.

💡 [c88a22] How does REPL state persist across process restarts?

Lean: event-replay. Each executed command is a ReplExecuted event. On resume, commands are replayed in order. Shell commands are sufficiently deterministic for this to work. Snapshot (binary blobs in events.jsonl) violates Contract Authority. Hybrid deferred.

💡 [835f01] Cost / determinism / debuggability

Real costs: PTY startup latency, per-command round-trip, replay non-determinism, harder trajectory debugging. For shell REPL: startup is fast; determinism is sufficient; ReplExecuted event log makes trajectories inspectable. For RLM-style llm_query recursion: additional LLM call budget analogues of max_depth and max_money_spent will be needed.

💡 [c86aa6] Interaction with strict resume

When REPL is built, the round-trip gate test must be extended to verify REPL state equivalence after resume. For shell REPL, "equivalent" means working directory and key environment variables match (not byte-for-byte stdout identity). This extension is a prerequisite for the REPL iteration, not for vanilla-subagents or handoff-automation work.

RQ-B questions

⚠ [7d2e8f] Is RQ-B even worth pursuing?

Kim et al., Cemri et al., the Anthropic engineering post, and the equal-budget parity finding (Tran & Kiela) all suggest caution. The counter-evidence (specialised phase agents, AdaptOrch) requires a careful task-decomposition design that may not generalise. This question should be revisited after RQ-D's seeded-handoff instrumentation provides real data on what Omega's actual subtask decomposition looks like in practice.

RQ-C questions

💡 [8e161a] How does an llm_query call relate to SessionId?

Lean: each llm_query invocation is its own session — own SessionId, own events.jsonl, own folder. Mirrors the existing decision that subagents are child sessions. Only the invocation surface changes (REPL function call vs. tool_use); the underlying spawn-a-child-session machinery is shared.

💡 [053c4a] What is "the session" in an RLM Omega?

Lean: today's framing survives intact. "The session" is the outer Omega instance the user interacts with. Its REPL is a property of that instance. Children spawned by llm_query are child sessions in exactly the same sense as vanilla subagents.

💡 [bab784] User interaction with an RLM Omega

Lean: chat messages stay; REPL is internal; UI optionally exposes REPL state as collapsible ReplExecuted event blocks. A dedicated "REPL console" pane is deferred.

💡 [1f28f7] Output format — autoregressive vs. variable

Lean: autoregressive prose at the user boundary; variable returns only within the REPL and in subagent-to-parent data flow. FINAL(variable)-style returns may apply within subagent calls but not to the user-facing reply.

RQ-D questions

💡 [e91b54] Seeded-handoff automation — how far to go first?

Three incremental steps (§6): trigger detection, seed-prompt drafting, dispatch. Each step is independently useful. The research question is: how much of the handoff does the agent need to automate before human review becomes the bottleneck rather than the safeguard? Empirical question; answer from instrumented use.

💡 [a3f7c2] Is the human-orchestrated, artefact-returning pattern worth formalising as a research object?

The pattern is distinct from both vanilla MAS and REPL-based RLM. It sidesteps the known failure modes. The literature has not studied it rigorously. Omega's session logs and commit history constitute a growing evidence base. The concrete research design: instrument the handoff events, measure outcomes, compare against sessions that did not use handoffs. Feasibility: high. Unique Omega advantage: high.
