Phase 1 SessionRef — Design Proposal

Status: Proposal — awaiting main-session review · Covers the nine open questions listed in docs/session-design-wip.md § "Before Phase 1" · No crate code is modified by this document.

Overview

Phase 1 introduces the identity primitives on which every subsequent cross-session feature depends. The decisions made here will be baked into events.jsonl forever; getting them right is more important than getting them fast.

⚠ Contract-authority reminder

Per the AGENTS.md Contract Authority rule, the Rust event types (OmegaEvent and friends in omega-types) are the canonical schema definition — the JSON files in events.jsonl are their serde projection. Once data is written to disk those files become a compatibility obligation: type changes must preserve deserialization of existing logs, or require explicit migration. Use serde attributes deliberately to honour that obligation — never defensively to suppress an error. The type system rejecting a malformed event is a feature. When in doubt, let it fail.

Each question below presents a concrete Recommendation callout followed by a collapsible Rationale section that explains the tradeoffs considered. Rust type definitions and JSON examples are included where relevant.

Q1 — SessionId and EventId as newtypes

1 Should SessionId and EventId be newtypes (pub struct SessionId(Uuid)) rather than plain type aliases (type SessionId = Uuid)?

✓ Recommendation — Yes, full newtypes

Define both as tuple structs wrapping uuid::Uuid with #[serde(transparent)]. Accept no alternative. 

// crates/omega-types/src/ids.rs

use serde::{Deserialize, Serialize};
use uuid::Uuid;

/// Unique identifier for an Omega session.  Wraps a UUID v7.
/// Newtype — prevents accidental mixing with `EventId` at compile time.
#[derive(Debug, Clone, PartialEq, Eq, Hash, Serialize, Deserialize)]
#[serde(transparent)]
pub struct SessionId(pub Uuid);

/// Unique identifier for a single event within any session.  Wraps a UUID v7.
/// Newtype — prevents accidental mixing with `SessionId` at compile time.
#[derive(Debug, Clone, PartialEq, Eq, Hash, Serialize, Deserialize)]
#[serde(transparent)]
pub struct EventId(pub Uuid);
Rationale (blast-radius, type safety, UUID v7)

F2 F13 Today SessionStartedEvent.session_id is a plain String equal to the folder name. Phase 1 replaces this with a genuine UUID. The newtype wrapper gives the compiler a second line of defence once that migration happens: passing a SessionId where an EventId is expected is a compile-time error, not a silent data corruption.

Why not a type alias? A type alias (type SessionId = Uuid) provides zero protection — the compiler treats it as the underlying type and allows interchanging the two freely. Given that these IDs will appear in event payloads that live on disk forever, the extra safety is mandatory for something with this blast radius.

UUID v7 is the right UUID variant here (time-ordered, which gives natural insertion order in any future index), consistent with the uuid::Uuid v7 dependency already planned in Phase 0.6.

#[serde(transparent)] ensures the JSON representation is just the UUID string ("018f..."), not a wrapped object ({"0": "018f..."}). This keeps events.jsonl readable and matches what readers expect.

The pub field inside the tuple struct is intentional: pattern matching (let SessionId(uuid) = id) is the idiomatic Rust way to unwrap without adding a dedicated accessor.

Q2 — One SessionRef type vs two types

2 Should there be a single SessionRef { session_id, event_id: Option<EventId> }, or two separate types — SessionRef (session-only) and EventRef (session + event)?

✓ Recommendation — Single SessionRef type

One type with event_id: Option<EventId>. None means "the session as a whole"; Some(eid) means a specific moment within the session. 

/// A reference to an Omega session, optionally pinned to a specific event.
///
/// - `event_id: None`    → refers to the session as a whole (e.g. a folder
///                          path, a subagent spawn before it has terminated).
/// - `event_id: Some(e)` → refers to a specific moment; `e` is the UUID of
///                          an event in `session_id`'s `events.jsonl`.
#[derive(Debug, Clone, PartialEq, Eq, Hash, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct SessionRef {
    pub session_id: SessionId,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub event_id: Option<EventId>,
}
Rationale (simplicity, optionality, two-type tradeoffs)

Against two types: the only distinction between a "session reference" and an "event reference" is whether the event ID is present. Encoding that as two separate Rust types would force every call site that might hold either to use an enum or a trait object, adding indirection for no real gain.

For optionality: the design already specifies concrete cases for both states. A SubagentSpawned event carries event_id: None (the child session exists but has no terminal event yet). A SubagentReturned event carries event_id: Some(terminal_in_child). Both fit naturally in one type.

Call-site clarity: code that requires an event-pinned ref enforces this with a runtime check or by constructing the ref only when an EventId is known. The Option is honest — it communicates "this may or may not be pinned to a moment" rather than hiding the distinction.

Future-proofing: if Phase 4 introduces forks that need a ref to a branch point, that branch point is an event — the same SessionRef{session_id, event_id: Some(fork_point)} shape works without schema change.

#[serde(skip_serializing_if = "Option::is_none")] means a session-only ref serialises as {"sessionId":"..."} rather than {"sessionId":"...","eventId":null}. This is cleaner in events.jsonl and does not violate the contract-authority rule because absence of the field is semantically equivalent to None (Serde's deserialize fills it in).

Q3 — Serde shape: struct vs encoded string

3 Should SessionRef serialise as a JSON struct ({"sessionId":"…","eventId":"…"}) or as a compact string ("<sid>:<eid>" / "<sid>")?

✓ Recommendation — Struct (camelCase fields)

Use the flat struct representation with #[serde(rename_all = "camelCase")], matching the existing events.jsonl field-naming convention. The string encoding is reserved for Display / logs / UI chips (see Q4). 

// Struct form — as it appears in events.jsonl

// Session-only reference (eventId absent):
{ "sessionId": "018f4c2e-3a1b-7d00-8000-abcdef012345" }

// Pinned-to-event reference:
{
  "sessionId": "018f4c2e-3a1b-7d00-8000-abcdef012345",
  "eventId":   "018f4c2f-1a2b-7e00-8000-fedcba987654"
}
Rationale (inspectability, grep-ability, contract authority)

Contract-authority argument: the on-disk log is the most public contract. Struct fields are self-describing and independently grep-able. A future developer can run grep sessionId events.jsonl and find every cross-session reference without knowing the string-encoding grammar.

Consistency: every existing event in events.jsonl uses camelCase struct fields (verified by the existing serde tests in omega-types/src/events.rs). A string-encoded SessionRef would be the only field that embeds structured data inside a string — an inconsistency that makes parsing harder for external tools.

String encoding has no advantages for persistence: the compactness argument only matters if refs appear in many positions where JSON verbosity is a real cost. In practice a SessionRef appears at most a handful of times per session (spawn + return). The extra ~50 bytes per field is irrelevant.

Embeddability: the struct form works equally well as a nested field in larger event payloads. No escaping or double-encoding is needed.

skip_serializing_if = "Option::is_none" on event_id keeps the session-only form clean. The deserialisation side uses Serde's implicit None default for absent optional fields — this is the correct (non-defensive) use of optionality: the field is genuinely optional in the domain.

Q4 — Display / FromStr format

4 What string format should SessionRef::to_string() produce, and how should FromStr parse it? Used in logs, error messages, and UI chips. Must round-trip.

✓ Recommendation — <sid> or <sid>#<eid>

Use a hash separator (#) between the session UUID and event UUID when the event ID is present. Session-only refs display as the bare UUID. Both SessionId and EventId also implement Display as their inner UUID string. 

use std::fmt;
use std::str::FromStr;

impl fmt::Display for SessionId {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "{}", self.0)
    }
}

impl fmt::Display for EventId {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "{}", self.0)
    }
}

impl fmt::Display for SessionRef {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match &self.event_id {
            None      => write!(f, "{}", self.session_id),
            Some(eid) => write!(f, "{}#{}", self.session_id, eid),
        }
    }
}

#[derive(Debug, PartialEq)]
pub struct SessionRefParseError;

impl fmt::Display for SessionRefParseError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "invalid SessionRef format; expected <uuid> or <uuid>#<uuid>")
    }
}

impl FromStr for SessionRef {
    type Err = SessionRefParseError;

    fn from_str(s: &str) -> Result<Self, Self::Err> {
        let parse_uuid = |u: &str| -> Result<Uuid, SessionRefParseError> {
            Uuid::parse_str(u).map_err(|_| SessionRefParseError)
        };
        match s.split_once('#') {
            None => {
                let sid = SessionId(parse_uuid(s)?);
                Ok(SessionRef { session_id: sid, event_id: None })
            }
            Some((sid_s, eid_s)) => {
                let sid = SessionId(parse_uuid(sid_s)?);
                let eid = EventId(parse_uuid(eid_s)?);
                Ok(SessionRef { session_id: sid, event_id: Some(eid) })
            }
        }
    }
}
Rationale (separator choice, round-trip, URL safety)

Why # as the separator?

  • UUID v7 strings contain only hex digits and hyphens — # never appears in a UUID, so split_once('#') is unambiguous.
  • The colon (:) was considered but is a reserved character in URIs, which matters if refs ever appear in URLs or command-line arguments without quoting.
  • The slash (/) was considered (matching session_id/event_id) but clashes with path segments if the ref is embedded in a URL route.
  • # reads naturally as "anchor" — the event is an anchor point within the session, matching the web/HTML metaphor without confusion.

Round-trip guarantee: FromStr parses exactly what Display emits, so ref.to_string().parse::<SessionRef>() is always Ok(ref). This is verified by a trivial property test.

Session-only refs display as bare UUIDs. This means folder names (which will be UUID-derived, see Q8) can be parsed as SessionRef directly when needed, without special-casing.

UI chips need not display the full UUID; a truncated prefix (018f4c2e…) is sufficient for human identification. The full string is used in logs and error messages where full traceability matters.

Q5 — Hash / Eq / Ord derives

5 Which comparison and hashing traits should SessionId, EventId, and SessionRef derive? Is Ord appropriate given UUID v7's time-ordered property?

✓ Recommendation — Derive Hash + Eq; do not derive Ord

Derive PartialEq + Eq + Hash on all three types. Do not derive PartialOrd or Ord on any of them. 

// SessionId and EventId:
#[derive(Debug, Clone, PartialEq, Eq, Hash, Serialize, Deserialize)]

// SessionRef:
#[derive(Debug, Clone, PartialEq, Eq, Hash, Serialize, Deserialize)]
Rationale (HashMap keying, Ord hazard, UUID v7 ordering)

Hash + Eq are required for the Phase 3.0 HashMap<SessionId, ActiveSession> that will replace the current Option<ActiveSession> in AppState.

Why no Ord? UUID v7 encodes a timestamp in the high bits, so UUID ordering is approximately time ordering. "Approximately" is the hazard: two UUIDs generated within the same millisecond may have an arbitrary sub-millisecond ordering, and two UUIDs from different machines have no guaranteed relationship. Exposing Ord on an ID type invites code that relies on comparison semantics that are not contractually guaranteed. If a sorted container is needed (e.g. a BTreeMap keyed by session), it should explicitly sort by the time field from the SessionStartedEvent, not by the ID itself.

Pragmatic consequence: omitting Ord makes the crate slightly less convenient for ad-hoc sorting, but eliminates an entire class of latent bugs. The contract-authority principle ("loud, not defensive") applies here too: a compilation error when someone tries to use id1 < id2 is preferable to silent time-ordering that breaks on multi-machine deployments.

SessionRef Hash / Eq delegate to the component fields' implementations, which is the correct structural equality: two refs are equal iff both their session_id and event_id fields are equal.

Q6 — Crate home

6 Which crate should own SessionId, EventId, and SessionRef?

✓ Recommendation — omega-types, in a new ids.rs module

All three types live in crates/omega-types/src/ids.rs, re-exported from the crate root alongside OmegaEvent. 

// crates/omega-types/src/lib.rs  (additions)
pub mod ids;
pub use ids::{EventId, SessionId, SessionRef};
Rationale (dependency graph, single source of truth)

omega-types is already the canonical home for OmegaEvent and all event structs — the types that every other crate (omega-agent, omega-server, omega-tools) imports. Adding IDs and refs here avoids creating a new dependency edge in the workspace graph.

Alternative considered: a separate omega-ids micro-crate. Rejected because the only gain would be slightly faster recompilation if IDs change without OmegaEvent changing. In practice these two change together (adding an EventId field to events), and a micro-crate adds workspace maintenance overhead with no architectural benefit.

A dedicated ids.rs module (rather than inlining into events.rs) keeps the types navigable as they grow — Origin, future RunId, etc. will all live there.

Q7 — EventId placement: envelope vs per-variant

7 Should EventId be placed on a common outer envelope (a header struct wrapping OmegaEvent), or on each variant struct individually? F3

✓ Recommendation — Outer envelope

Introduce a thin LoggedEvent envelope that carries event_id: EventId alongside the existing OmegaEvent. The envelope is what gets written to and read from events.jsonl. 

/// The on-disk unit stored in `events.jsonl`.
///
/// Every persisted event is wrapped in this envelope.  The inner
/// `OmegaEvent` is flattened into the same JSON object so that the
/// `"type"` discriminator remains a top-level key.
///
/// Reading old log lines that predate the envelope (no `"eventId"` field):
/// `event_id` defaults to `None`.  This is the ONLY correct use of a serde
/// default here — old events genuinely have no ID.  Do not add more defaults.
#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct LoggedEvent {
    /// Stable identity for this event.  `None` only for pre-Phase-1 log lines.
    #[serde(skip_serializing_if = "Option::is_none")]
    pub event_id: Option<EventId>,

    /// The event payload.  Flattened so `"type"` remains a top-level key.
    #[serde(flatten)]
    pub event: OmegaEvent,
}

At write time the event store always assigns a fresh UUID v7: 

// In EventStore::append (sketch):
let logged = LoggedEvent {
    event_id: Some(EventId(Uuid::now_v7())),
    event,
};
writeln!(file, "{}", serde_json::to_string(&logged)?)?;
Rationale (F3, DRY, flat JSON, old-log compatibility)

F3 The audit notes that events have no stable IDs today — they're identified by their position in the file. Adding an ID to every variant struct individually would require touching ~20 structs in events.rs with identical boilerplate, and every new variant would silently omit the ID if the author forgets.

The envelope pattern solves this once: the ID is added in one place, assigned by the writer (not the event constructor), and the rest of the codebase continues constructing OmegaEvent variants without caring about IDs.

#[serde(flatten)] keeps the JSON representation flat — the "type" discriminator stays at the top level and existing log readers continue to work: 

// A post-Phase-1 line in events.jsonl:
{
  "eventId": "018f4c2f-1a2b-7e00-8000-fedcba987654",
  "type":    "user_message",
  "time":    "2025-07-01T10:00:00.000Z",
  "content": "Hello"
}

Old-log compatibility: when deserialising a line written before Phase 1 (no "eventId" field), event_id defaults to None. This is the explicitly correct, non-defensive use of skip_serializing_if + implicit Option default: old events genuinely have no ID. The contract-authority rule's "loud, not defensive" principle is honoured — any other absent field in the post-Phase-1 schema will cause a deserialization error.

The per-variant alternative (adding event_id: EventId to every struct) was considered. It would make the ID visible in the type system at every event-construction site, which is marginally safer. But it forces callers to generate an ID even when constructing an event for an in-memory purpose that won't be persisted (e.g. unit-test construction, WS broadcasting of an already-logged event). The envelope cleanly separates "event payload" from "event identity in the log".

Q8 — Bare SessionId vs SessionRef

8 When should a call site use the bare SessionId vs the full SessionRef? The design doc mentions folder names take bare IDs but event payloads take SessionRef — establish the rule.

✓ Recommendation — Bare ID for identity/indexing; SessionRef for cross-session event payloads

The rule is: use SessionId wherever you name a session as a session; use SessionRef wherever an event payload points from one session into another.

ContextTypeReason
Session folder name / directory hint SessionId The folder is the session; no event pinning needed
HashMap<SessionId, ActiveSession> key SessionId Index into a live session registry; moment is always "now"
SessionStartedEvent.session_id SessionId Declares this session's own identity; not a cross-session reference
SubagentSpawned.child SessionRef (event_id: None) Parent's event stream points into a child session (whole session)
SubagentReturned.child SessionRef (event_id: Some(_)) Points to the terminal event in the child
Origin::SubagentOf { parent } SessionRef (event_id: Some(_)) Child session records the spawn event in the parent (see Q9)
SessionMetadata.resumed_from F5 SessionRef (event_id: None) Soft-resume points at the predecessor session as a whole
Resolving a ref to a folder on disk Extracts session_id from SessionRef Resolution uses the bare ID to find the folder
Rationale (F5, soft-resume migration, identity vs pointer)

The core intuition is:

  • A bare SessionId says "I am talking about this session's identity." It is appropriate when the session itself is the subject.
  • A SessionRef says "I am pointing from here to something over there." It is appropriate in event payloads that create a navigable link across session boundaries.

F5 Today SessionMetadata.resumed_from and the two resume events use folder-name strings. Under Phase 1, these become SessionRef with event_id: None (the whole predecessor session is the reference, not a particular moment in it). The migration is: when upgrading old sessions, resolve the folder name to a SessionId and construct the ref from there.

Why not always SessionRef? Using SessionRef as a HashMap key is awkward and semantically wrong — a live session is keyed by identity, not by a pointer into it. Keeping SessionId for identity use avoids a class of bugs where a session-only ref and an event-pinned ref with the same session_id would hash differently.

Q9 — Origin enum exact shape

9 What is the exact shape of the Origin enum? Specifically, in SubagentOf { parent: SessionRef }, should parent.event_id be Some(_) (the spawn event in the parent) or None (just the parent session)?

✓ Recommendation — Require event_id: Some(_); enforce at construction

Origin::SubagentOf { parent: SessionRef } must carry the spawn event ID from the parent (event_id: Some(spawn_event_id)). A session-only ref (event_id: None) is semantically wrong here and should be rejected at construction time. 

/// Records how this session came to exist.
///
/// Stored in `SessionMetadata` (session.jsonc) and consulted by the
/// UI to build the session tree.
///
/// Extensible: `ForkOf` is planned for Phase 4 but not added now.
/// Unknown variants deserialise to `Unknown` so new variants in
/// newer Omega versions degrade gracefully rather than panicking.
#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
#[serde(tag = "type", rename_all = "snake_case")]
#[non_exhaustive]
pub enum Origin {
    /// Top-level session: started directly by the user.
    Root,

    /// This session was spawned as a subagent by the given parent session.
    ///
    /// `parent.event_id` is `Some(spawn_event)` — the UUID of the
    /// `SubagentSpawned` event in the parent's `events.jsonl`.  This is
    /// contractually required; a `None` event_id is a construction error.
    SubagentOf { parent: SessionRef },
}

impl Origin {
    /// Constructs a `SubagentOf` variant, enforcing that the parent ref
    /// carries a concrete spawn-event ID.
    ///
    /// # Panics
    /// Panics if `parent.event_id` is `None`.  Callers must supply the
    /// spawn event ID from the parent session.
    pub fn subagent_of(parent: SessionRef) -> Self {
        assert!(
            parent.event_id.is_some(),
            "SubagentOf requires a concrete spawn-event ID in parent.event_id"
        );
        Self::SubagentOf { parent }
    }
}
Rationale (transparency requirement, navigability, F5 analogy)

The transparency requirement. The design document states: "Every event that references another session or event must, from day one: render visibly in the feed; show the literal IDs; have a 'more' button revealing full JSON; ideally be clickable to navigate." For that navigation to work, the child's session.jsonc must carry a direct link back to the exact event in the parent that caused the spawn. A session-only ref (event_id: None) gives you the parent session but not the spawn moment, making the back-link less precise.

Audit analogy (F5). The audit notes that current cross-session references use folder names (coarse). Phase 1 replaces these with UUID refs precisely because coarse references lose navigability. A SubagentOf with no event ID repeats the same mistake at a finer level.

When is the spawn event ID known? At the moment the parent writes the SubagentSpawned event, it receives the EventId from the event store. This ID is then passed as part of the seed to the subagent's session metadata. The timing is clear: the child is created after the event is written, so the ID is always available.

Enforcing Some(_). Rather than encoding the invariant in the type (which would require a second EventRef type — the two-type approach rejected in Q2), we use a constructor that panics on None. This is the "loud" approach: a programming error is caught at construction time during development, not silently at runtime or in production logs. The serde deserialise path does not enforce this (old logs may theoretically lack the ID), but write-time construction always goes through Origin::subagent_of.

#[non_exhaustive] ensures that when Phase 4 adds ForkOf, any match on Origin in other crates will produce a compile error rather than silently hitting a _ => unreachable!() arm. This is the "loud, not defensive" principle applied to enum evolution.

Serde tag. #[serde(tag = "type")] produces {"type":"root"} and {"type":"subagent_of","parent":{...}}, consistent with the existing tagged-enum convention in OmegaEvent.

Full type summary

All types in one place for reference. These are the definitions to implement in Phase 1.1–1.3.

// crates/omega-types/src/ids.rs
// ============================================================
// Omega identity primitives — Phase 1.1–1.3
// ============================================================
//
// Contract-authority: these types are the Rust projection of the
// on-disk format.  Changes here are breaking changes to events.jsonl.

use std::fmt;
use std::str::FromStr;
use serde::{Deserialize, Serialize};
use uuid::Uuid;

// ── Leaf ID types ───────────────────────────────────────────

#[derive(Debug, Clone, PartialEq, Eq, Hash, Serialize, Deserialize)]
#[serde(transparent)]
pub struct SessionId(pub Uuid);

#[derive(Debug, Clone, PartialEq, Eq, Hash, Serialize, Deserialize)]
#[serde(transparent)]
pub struct EventId(pub Uuid);

// ── SessionRef ──────────────────────────────────────────────

#[derive(Debug, Clone, PartialEq, Eq, Hash, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct SessionRef {
    pub session_id: SessionId,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub event_id: Option<EventId>,
}

impl SessionRef {
    pub fn session(session_id: SessionId) -> Self {
        Self { session_id, event_id: None }
    }

    pub fn event(session_id: SessionId, event_id: EventId) -> Self {
        Self { session_id, event_id: Some(event_id) }
    }
}

// ── Display / FromStr ───────────────────────────────────────

impl fmt::Display for SessionId {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "{}", self.0)
    }
}

impl fmt::Display for EventId {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "{}", self.0)
    }
}

impl fmt::Display for SessionRef {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match &self.event_id {
            None      => write!(f, "{}", self.session_id),
            Some(eid) => write!(f, "{}#{}", self.session_id, eid),
        }
    }
}

#[derive(Debug, PartialEq)]
pub struct SessionRefParseError;

impl fmt::Display for SessionRefParseError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "invalid SessionRef: expected <uuid> or <uuid>#<uuid>")
    }
}

impl FromStr for SessionRef {
    type Err = SessionRefParseError;

    fn from_str(s: &str) -> Result<Self, Self::Err> {
        let pu = |u: &str| Uuid::parse_str(u).map_err(|_| SessionRefParseError);
        match s.split_once('#') {
            None              => Ok(Self::session(SessionId(pu(s)?))),
            Some((sid, eid))  => Ok(Self::event(SessionId(pu(sid)?), EventId(pu(eid)?))),
        }
    }
}

// ── Origin ──────────────────────────────────────────────────

#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
#[serde(tag = "type", rename_all = "snake_case")]
#[non_exhaustive]
pub enum Origin {
    Root,
    /// parent.event_id must be Some(_); use Origin::subagent_of() to construct.
    SubagentOf { parent: SessionRef },
}

impl Origin {
    pub fn subagent_of(parent: SessionRef) -> Self {
        assert!(parent.event_id.is_some(),
            "SubagentOf requires a concrete spawn-event ID");
        Self::SubagentOf { parent }
    }
}

Display format examples

// Session-only ref:
let r = SessionRef::session(SessionId(uuid_a));
assert_eq!(r.to_string(), "018f4c2e-3a1b-7d00-8000-abcdef012345");

// Event-pinned ref:
let r = SessionRef::event(SessionId(uuid_a), EventId(uuid_b));
assert_eq!(r.to_string(), "018f4c2e-3a1b-7d00-8000-abcdef012345#018f4c2f-1a2b-7e00-8000-fedcba987654");

// Round-trip:
let s = r.to_string();
assert_eq!(s.parse::<SessionRef>().unwrap(), r);
