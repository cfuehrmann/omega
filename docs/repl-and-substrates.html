Phase 3 — Multi-language substrate

  • 3.1 — Pluggable REPL backend abstraction. Python REPL becomes one implementation of a ReplBackend trait (or equivalent). Abstraction covers: process spawn, sentinel protocol, SIGINT/kill-group escalation, tee-always output, bootstrap.
  • 3.2 — Bash REPL as second implementation. Same plumbing as Python REPL. No bootstrap required. Tool name: bash_repl.
  • 3.3 — Tool selection includes backend choice. The per-tool checkbox grid from Phase 2.1 naturally covers this — no extra mechanism needed.

Overall acceptance: "user-grade REPL"

  • User starts a session with REPL enabled via a UI gesture — no env-var fiddling.
  • REPL code and output render clearly in the feed.
  • User can recognize, from the UI alone, when REPL was invoked, what code ran, what came back, and whether anything went wrong.
  • Python REPL is robust enough for real work (long text analysis, log inspection) without expert hand-holding.

🧭 Decisions on record

  1. Q1 — Tool selection is start-only. Captured in SessionStartedEvent, immutable for the lifetime of the session. Caching hygiene; symmetry with model/effort/system-prompt; avoids growing the domain-event surface. Users change toolset by starting a new session.
  2. Q2 — Per-tool checkboxes as the UI model. Maximum granularity: one checkbox per tool. "REPL mode / Tier 1 / Tier 2" become saved configurations (named subsets) users can pick from — strictly more powerful than tiered abstractions.
  3. Q3 — Migrate env vars into the tool-selection mechanism. No env-var carve-out unless a feature genuinely cannot map to a tool selection (none identified). Tool selection is the new mechanism.
  4. Q4 — Python first, Bash REPL second; Steel and PowerShell deferred. Bash REPL is the second substrate because it isolates the statefulness variable from the substrate-switching variable — the cleanest experimental design. Steel and PowerShell are candidates documented in the substrates table but deferred until the abstraction shakes out.
  5. Strict session resume — dropped (with code deleted). Full reasoning at session-design.html §decision-drop-strict-resume. Short form: the observational-equivalence promise never survived world drift across resumes; the user's empirical usage data does not motivate the complexity; AI-based forensics over events.jsonl covers the real use case.
  6. AI-resume kept; sunset when subagents land. The LLM-summary resume mechanism is operationally useful and honest about its semantics (it summarizes a parent session and seeds a new session with that summary as preamble — no equivalence claim). It stays until the subagent / prompt-passing feature subsumes its use case. See session-design.html §ai-resume-sunset.
  7. View-only WS endpoint — deferred indefinitely. Documented in session-design.html §oq-repl-view. AI-based forensics over events.jsonl covers the actual use case. Reopen only if a concrete UI need surfaces.
  8. Fork-from-event primitive — not planned. The existing AI-resume already implements this in spirit (new session seeded from a parent's events digest). Recorded so the option isn't forgotten if a concrete additional use case appears.

The pivot — how we got from benchmarks to user-grade

Between v0.1.8 and v0.1.14, a sequence of benchmark experiments on two cached Terminal-Bench 2 tasks (fix-git and crack-7z-hash) hardened the Python REPL from an MVP into a capable tool: bootstrap-on-missing, per-call SIGINT/kill-group escalation, tee forensics, sentinel renaming, and system prompt refinements were all diagnosed from failing runs and fixed iteratively. The v0.1.13 Tier 2 result — 26/26 tool calls via python_repl, genuine cross-call state, Python threading, three timeout recoveries, task passed in 10 min 44 s — validated that the approach works in practice.

After that result, the direction pivoted from benchmark-driven experimentation to user-grade implementation. The engineering question is no longer "does REPL help on Terminal-Bench?" but "how do we make the REPL useful and visible to a real user, without env-var fiddling, in a way that is obviously correct from the UI alone?" Four design decisions (Q1–Q4 above) were settled during the pivot discussion and are encoded directly into the three-phase plan.

(a) Standard-budget timeout — path-tracing (1800 s)

Session ran from 2026-05-30T22:56:10Z to 2026-05-30T23:24:06Z (27 min 56 s, i.e. right at the 1800 s wall). The agent made exactly 56 tool calls, every single one a python_repl call. No other tool was called — the agent never called write_file, run_command, or any file-writing primitive. The task asks the agent to write a C program (image.c) that reproduces a ray-traced image to within 0.99 L2 similarity. The entire session was spent reverse-engineering the scene geometry by numerical analysis of the provided PPM pixels:

3 · NonZeroAgentExitCodeError triage

No events.jsonl exists in either trial's agent directory — both processes exited before the omega session was created. The root causes come from exception.txt and the captured stdout in each trial.log.

(a) fix-code-vulnerability — exit code 1, wall time ≈ 45 s

The omega CLI printed to stdout before exiting:

omega: there are uncommitted changes in the working tree. Commit or stash them before running omega, or pass --allow-dirty to proceed anyway.

The bench wrapper runs cd /app && omega run --instruction …. The fix-code-vulnerability Docker image places the Bottle repository at /app and that directory is a live git repo with modified (pre-seeded) files. omega's startup check detects the dirty working tree and refuses to proceed. The bench wrapper does not pass --allow-dirty. The ≈40 s before the exit is accounted for by the apt-get update && install ca-certificates setup step that runs before the omega binary is invoked.

Root cause: omega-cli startup check triggered by a task image whose /app directory is a git repository with uncommitted files. Other tasks either have a clean /app git repo or no git repo at /app at all, so they are unaffected.

Recommendation: Open an issue: the bench wrapper should pass --allow-dirty unconditionally, or omega should default to allow-dirty when no session journal exists for the path. Do not fix here.

(b) pytorch-model-recovery — exit code 2, wall time ≈ 336 s

The omega CLI printed to stdout before exiting:

error: unexpected argument '- ' found
Usage: omega run [OPTIONS] --instruction <INSTRUCTION>

The task instruction text begins with - You are given a PyTorch state dictionary… (leading dash-space, a YAML-style list bullet). The bench wrapper passes the instruction as --instruction {shlex.quote(instruction)}. Python's shlex.quote single-quotes the value; the shell strips the quotes and passes the raw string to the binary. clap (the Rust CLI parser) sees a value beginning with - after --instruction and, depending on the clap version and argument definition, may treat it as an unrecognised flag rather than a plain string value. Exit code 2 is clap's standard parse-error exit. The ≈5.5-minute wall time before the exit is accounted for by container setup and the model-weight file loading that happens as part of the Docker image startup.

Root cause: omega-cli's argument parser rejects instruction strings beginning with - . A single-character leading dash is sufficient to trigger the parse error; this is a clap argument-parsing edge-case.

Recommendation: Open an issue: the bench wrapper should use --instruction=<value> (with =) instead of --instruction <value> (space-separated) so that leading-dash instruction strings cannot be misinterpreted by clap. Alternatively, omega should declare --instruction with allow_hyphen_values(true). Do not fix here.

Same or different root cause?

Different. (a) is an omega startup policy issue (dirty-tree guard); (b) is an omega CLI parsing issue (leading-dash instruction). Both are omega-cli bugs triggered by unusual task-environment conditions; neither is a harbor-infra or GLIBC issue. Neither is expected to be widespread — only tasks whose /app is a dirty git repo would hit (a), and only tasks whose instruction literally starts with - would hit (b). Both are one-line fixes in the bench wrapper.

F-B Additive REPL does not engage the LLM

On both tasks, when python_repl was offered alongside the full standard toolset, the LLM never called it. Both tasks passed via normal routes. The additive design cannot measure REPL value because the experimental variable is never engaged. Consistent with Kim et al.'s Baseline Paradox (see §2 and subagents-research.html §2). Any benchmark of REPL value requires a constrained treatment arm.

F-C Tier 1 engages REPL on file-centric tasks; inert on shell-centric ones

With file tools removed, the LLM used python_repl to write the conflict-resolved file on fix-git (task passed). On crack-7z-hash the removed tools were inert for the task (it is a password-cracking task, not a file-editing task), so the LLM timed out. N=1 each.

Detail — what the python_repl call on fix-git looked like

The LLM resolved a Git merge conflict by writing the final file in a single python_repl call using a Python string literal and open().write(). Clean signal: file-write tools were absent; Python was the natural fallback. Event logs: bench/jobs/v019-inspect-fix-git-limit-mode/.

F-D Tier 2 first passed via a fetch_url.postprocess shell backdoor

The first Tier 2 run (v0.1.10) produced a pass for the wrong reason: the LLM used fetch_url.postprocess as a shell execution backdoor (31 of 33 calls). After closing the loophole (removing postprocess when shell tools are gated), the next run (v0.1.11) failed — marking the start of real Tier 2 experimentation.

F-E Several rounds of python_repl hardening were needed

After closing the loophole: v0.1.11 failed (no bootstrap), v0.1.12 failed (no timeout recovery). The hardening work — bootstrap-on-missing, per-call 60 s timeout with SIGINT/kill-group escalation, tee-always forensics, tail-bias on timeout, system-prompt variable-pattern guidance — accumulated across v0.1.11–v0.1.13 and is now part of the production codebase.

Detail — the hardening sequence
  • Sentinel rename (REPL_END → REPL_RESPONSE): original sentinel appeared in Python tracebacks, causing premature truncation.
  • Bootstrap-on-missing: benchmark containers lack Python3 on PATH at startup; added apt-get install -y python3 on first call with a python_repl_bootstrapped event on success.
  • Per-call timeout + escalation: 60 s default; sends SIGINT first (allowing Python to print partial output), escalates to process-group kill. REPL state is preserved across the interruption.
  • tee-always + tail-bias: all output written to a side-channel log; on timeout the last N bytes returned so the LLM sees where execution was when interrupted.
  • System-prompt variable-pattern guidance: updated to suggest storing results in variables so cross-call state accumulates naturally.

F-F v0.1.13 Tier 2 on crack-7z-hash: 100% REPL, genuine state, 10 min 44 s — passed

26/26 tool calls were python_repl, the Python interpreter carried state across multiple variable chains (proc, hash_val, passwords, found_password), the LLM used Python threading to parallelise the password search, and three subprocess timeouts were recovered from with REPL state intact. Task passed in 10 m 44 s total. N=1, but the qualitative trajectory is clean. Cost ($0.149) is competitive with sonnet baselines on the same task (range $0.076–$0.403), though not the cheapest pass.

🔬 Methodological note: the language-choice question

⚠ Retraction — earlier framing used cargo-culted language essentialism

Phrases like "orchestration is shell-shaped" or "Python is a natural fit for coding work" presuppose that languages have intrinsic domains they are for. This conflicts with Turing equivalence and conflates three empirically independent factors. The framing below is the correction.

The decomposed empirical framing

The actual phenomenon has three independent axes — none of which is "what language X is for":

  • Encoding cost c(L, P): the number of tokens needed to express operational pattern P in language L. Shell has very low encoding cost for pipe-composition; Python has low encoding cost for in-process data manipulation. Both are measurable.
  • Model fluency f(M, L, P): how reliably model M produces correct, idiomatic L-code for pattern P. A property of M's training distribution. Current frontier models have very high fluency in Python and Bash; much lower in Nushell or Steel.
  • Pattern density: how much of a benchmark's mass falls on pattern P versus other patterns.

Total cost is the sum over patterns of c(L, P) × density(P) / f(M, L, P). Lower is better. The language that minimises this sum for a given task and model is the better substrate — empirically, model-dependent, and contingent on the task distribution. No language-domain essentialism.

The Felleisen-expressiveness sharpening — why shell's apparent win is about encoding density, not expressive power

Shell's pipe-composition operator is macro-expansible into Python; the reverse is not true. Python is strictly more expressive in Felleisen's formal sense (1991): it can express patterns that shell cannot without a global program transformation. Shell's apparent win on orchestration-heavy tasks is therefore not about expressive power — it is about encoding density for chained subprocess invocation.

Ruby's backtick syntax (`cmd`) demonstrates the point: shell-level encoding density is achievable as syntax sugar inside a more expressive language. The sh() helper is the minimal instance of this for Python.

Why coding-agent work looks shell-aligned — without essentialism

Coding-agent work consists largely of stateful operations on a file tree via specialised external programs: compile, test, lint, format, grep, git, diff, etc. Those programs are CLI tools designed for shell. That is the load-bearing fact — contingent on Unix history, not on any intrinsic property of programming or shell. A future ecosystem with typed library APIs (pygit2, LSP clients, etc.) would shift the encoding-cost balance toward Python. The framing is stable; only the numbers would change.

⚠ Maintenance burden of structured tool wrappers

A library wrapping git, cargo, grep, etc. with typed APIs requires permanent maintenance: every subcommand, every flag, every output-format change ships a breakage. The sh() helper as minimum-viable experiment

The proposed preamble addition: 

def sh(cmd: str, *, timeout: float | None = None) -> tuple[str, str, int]:
    r = subprocess.run(["bash", "-c", cmd], capture_output=True,
                       text=True, timeout=timeout)
    return r.stdout, r.stderr, r.returncode
