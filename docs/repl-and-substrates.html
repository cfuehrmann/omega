Between v0.1.8 and v0.1.14, a sequence of benchmark experiments on two cached Terminal-Bench 2 tasks hardened the Python REPL from an MVP into a capable tool: bootstrap-on-missing, per-call SIGINT/kill-group escalation, tee forensics, sentinel renaming, and system-prompt refinements were all diagnosed from failing runs and fixed iteratively. The v0.1.13 Tier 2 result — 26/26 tool calls via python_repl, genuine cross-call state, Python threading, three timeout recoveries, task passed in 10 min 44 s — validated that the approach works in practice.

The direction now pivots from benchmark-driven experimentation to user-grade implementation. The benchmark question (does REPL help on Terminal-Bench?) is no longer the primary driver. The engineering question is: how do we make the REPL useful and visible to a real user, without env-var fiddling, in a way that is obviously correct from the UI alone?

Four design decisions were settled during the pivot discussion (Q1–Q4 below) and are encoded directly into the three-phase plan.

Phase 1 Cleanup and consolidation

✓ Q3 — Migrate env vars into tool-selection mechanism

The three REPL feature flags (OMEGA_FEATURE_REPL, OMEGA_FEATURE_REPL_REPLACES_FILEOPS, OMEGA_FEATURE_REPL_REPLACES_SHELL) are migrated into a tool-selection list captured at session creation in SessionStartedEvent. Default = current full toolset. The env vars are removed. System-prompt assembly is updated to reflect the actual toolset; the "REPL mode / Tier 1 / Tier 2" terminology is dropped. Tool selection IS the new mechanism; no env-var carve-out unless a feature genuinely does not map to a tool selection (none identified).

  • 1.1 — Refactor python_repl.rs. The file had grown to ~1600 lines through layered additions. File split landed at 8f557aa: the single file is now a directory module (python_repl/{wrapper,sentinel,tee,output,bootstrap,repl}.rs), visibility narrowed, tests + mutants green. One small follow-up remains: extract a shared read_phase helper from the three near-duplicate I/O loops in repl.rs::execute() (primary read, soft-grace, hard-drain). Behavior-preserving; the helper must respect the drain loop's deliberate lack of sentinel check.
  • 1.2 — Design and implement tool-selection mechanism. Add a tool_selection (or equivalent) field to SessionStartedEvent. Remove the three REPL env vars. Update system-prompt assembly to derive toolset from the event field rather than env vars. Update domain snapshot and strict resume to restore the toolset on resume. Update WS protocol so the session-creation message carries the tool selection.
  • 1.3 — Add sh() helper to python_repl preamble. Pre-load def sh(cmd, *, timeout=None) in the Python startup code; mention it in the system-prompt addendum. This closes the encoding-cost gap with bare shell on subprocess-heavy tasks (see § sh() helper). The [b3e7f2] benchmark experiment is folded into Phase 1 as an ergonomic improvement rather than a separate research step.

Phase 2 User-grade affordances

✓ Q1 — Tool selection is start-only

Tool selection is captured in SessionStartedEvent and is immutable for the lifetime of the session. Mid-session toggling is not supported. Reason: caching hygiene (the system prompt is stable across turns); symmetry with model/effort/system-prompt which are all start-time decisions; avoids growing the domain-event surface. Users change their tool set by starting a new session.

✓ Q2 — Per-tool checkboxes as the UI model

Maximum granularity: one checkbox per tool. The old "REPL mode / Tier 1 / Tier 2" abstractions become saved configurations that users can pick from ("default", "REPL-only", "REPL no-shell", etc.). This is strictly more powerful — saved configurations are just named subsets of per-tool toggles, with no loss of flexibility.

  • 2.1 — Tool-selection UI at session creation. Per-tool checkboxes in the session-creation panel. Small list of preset configurations (default, REPL-only, REPL no-shell, etc.) as quick-picks above the checkbox grid.
  • 2.2 — REPL observability in the UI feed. Best-possible rendering of REPL events: code block visible, output with truncation marker, bootstrap event surfaced, timeout/escalation event surfaced. Consider dedicated REPL-call rendering if low implementation cost.
  • 2.3 — Strict session resume for REPL sessions stays deferred (subject to revision). Current disposition: strict_resume continues to refuse REPL sessions loudly (StrictResumeError::ReplResumeUnsupported). The view-only WS path [oq-repl-view] and the event-replay design [oq-repl-replay] (both in docs/session-design.html) remain documented but unimplemented. For Phase 2, starting a new session is the resume story. Open: the broader strict-resume policy is under reconsideration; once that lands, this disposition is revisited.

Phase 3 Multi-language substrate

✓ Q4 — Python first, Bash REPL second; Steel/PowerShell deferred

Bash REPL is the second substrate because it isolates the statefulness variable from the substrate-switching variable — the cleanest experimental design. Bash was already available (as stateless run_command); adding a stateful Bash REPL lets us measure exactly what statefulness contributes. Steel and PowerShell are documented candidates in the candidate substrates table but are deferred until the abstraction is shaken out.

  • 3.1 — Pluggable REPL backend abstraction. Python REPL becomes one implementation of a ReplBackend trait (or equivalent). The abstraction covers: process spawn, sentinel protocol, SIGINT/kill-group escalation, tee-always output, bootstrap (where applicable).
  • 3.2 — Bash REPL as second implementation. Same plumbing as Python REPL. No bootstrap required (bash is always present). Tool name: bash_repl.
  • 3.3 — Tool selection includes REPL backend choice. The per-tool checkbox grid from Phase 2 naturally covers this: enabling python_repl vs. bash_repl vs. both.
  • Steel and PowerShell remain documented in the substrates table; no Phase 3 implementation.

Acceptance criteria for "user-grade REPL"

  • User starts a session with REPL enabled via a UI gesture — no env-var fiddling.
  • REPL code and output render clearly in the feed.
  • User can recognize, from the UI alone, when REPL was invoked, what code ran, what came back, and whether anything went wrong.
  • Python REPL is robust enough for real work (long text analysis, log inspection) without expert hand-holding.

Phase 1.1 follow-up — extract read_phase helper (✓ done, 7cd3bd6)

Historical brief preserved for context; the work landed at 7cd3bd6.

The file split has landed (8f557aa); the file is now a directory module under crates/omega-tools/src/python_repl/. One mechanical cleanup remains: in python_repl/repl.rs, execute() contains three near-duplicate I/O loops — primary read, soft-grace, hard-drain — that share 80% of their structure (pin a timeout future, tokio::select! between read_line and the timeout, tee each line, decide termination). Extract a private helper with signature roughly: 

enum ReadStop { Sentinel, Eof, Error(String), Timeout }
async fn read_phase(
    &mut self,
    timeout: Duration,
    lines: &mut Vec<String>,
    log: &mut Option<BufWriter<File>>,
    check_sentinel: bool,
) -> ReadStop { ... }

Behavior-preservation constraint. The three loops have real differences: primary returns an "early exit" error message on EOF/Err; grace breaks quietly on EOF/Err; drain does not check the sentinel and just tees until EOF or drain timeout. The check_sentinel: bool parameter is what distinguishes them. Do not accidentally regularize the drain loop to check the sentinel — that would be a semantic change. If the lack of sentinel check in drain feels like a latent bug, raise it as a question rather than fixing it silently.

📊 Findings to date

Between v0.1.8 and v0.1.14 we ran a sequence of feature-flag benchmark experiments on two cached Terminal-Bench 2 tasks (fix-git and crack-7z-hash). The hardening that accumulated across those versions — bootstrap-on-missing, SIGINT/kill-group escalation, tee forensics, sentinel renaming, system-prompt refinements — was necessary to make constrained-tool operation a real experiment. The current codebase captures all of these lessons; the iteration is complete. Details of individual version increments are in the collapsible in §7.

F-A Feature-flag tier matrix is sound

Three independent boolean flags (recorded in SessionStartedEvent.features) produced the expected toolset across all valid combinations. All four combinations (with repl=1) were exercised end-to-end. The matrix enables apples-to-apples comparisons without rebuilds. Phase 1 migrates this mechanism from env vars to a tool-selection list in the event schema.

F-B Additive REPL does not engage the LLM

On both tasks, when python_repl was offered alongside the full standard toolset, the LLM never called it. Both tasks passed via normal routes. The additive design cannot measure REPL value because the experimental variable is never engaged. Consistent with Kim et al.'s Baseline Paradox (see §2 and subagents-research.html §2). Any benchmark of REPL value requires a constrained treatment arm.

F-C Tier 1 engages REPL on file-centric tasks; inert on shell-centric ones

With file tools removed, the LLM used python_repl to write the conflict-resolved file on fix-git (task passed). On crack-7z-hash the removed tools were inert for the task (it is a password-cracking task, not a file-editing task), so the LLM timed out. N=1 each.

Detail — what the python_repl call on fix-git looked like

The LLM resolved a Git merge conflict by writing the final file in a single python_repl call using a Python string literal and open().write(). Clean signal: file-write tools were absent; Python was the natural fallback. Event logs: bench/jobs/v019-inspect-fix-git-limit-mode/.

F-D Tier 2 first passed via a fetch_url.postprocess shell backdoor

The first Tier 2 run (v0.1.10) produced a pass for the wrong reason: the LLM used fetch_url.postprocess as a shell execution backdoor (31 of 33 calls). After closing the loophole (removing postprocess when shell tools are gated), the next run (v0.1.11) failed — marking the start of real Tier 2 experimentation.

F-E Several rounds of python_repl hardening were needed

After closing the loophole: v0.1.11 failed (no bootstrap), v0.1.12 failed (no timeout recovery). The hardening work — bootstrap-on-missing, per-call 60 s timeout with SIGINT/kill-group escalation, tee-always forensics, tail-bias on timeout, system-prompt variable-pattern guidance — accumulated across v0.1.11–v0.1.13 and is now part of the production codebase.

Detail — the hardening sequence
  • Sentinel rename (REPL_END → REPL_RESPONSE): original sentinel appeared in Python tracebacks, causing premature truncation.
  • Bootstrap-on-missing: benchmark containers lack Python3 on PATH at startup; added apt-get install -y python3 on first call with a python_repl_bootstrapped event on success.
  • Per-call timeout + escalation: 60 s default; sends SIGINT first (allowing Python to print partial output), escalates to process-group kill. REPL state is preserved across the interruption.
  • tee-always + tail-bias: all output written to a side-channel log; on timeout the last N bytes returned so the LLM sees where execution was when interrupted.
  • System-prompt variable-pattern guidance: updated to suggest storing results in variables so cross-call state accumulates naturally.

F-F v0.1.13 Tier 2 on crack-7z-hash: 100% REPL, genuine state, 10 min 44 s — passed

26/26 tool calls were python_repl, the Python interpreter carried state across multiple variable chains (proc, hash_val, passwords, found_password), the LLM used Python threading to parallelise the password search, and three subprocess timeouts were recovered from with REPL state intact. Task passed in 10 m 44 s total. N=1, but the qualitative trajectory is clean. Cost ($0.149) is competitive with sonnet baselines on the same task (range $0.076–$0.403), though not the cheapest pass.

F-G Inner/outer timeout misalignment in v0.1.13's three SIGINT escalations

All three escalations arose from misalignment between the LLM's inner timeouts (subprocess.run(..., timeout=N) or time.sleep(N)) and the python_repl outer timeout (60 s default): no inner bound at all (#3), equal inner/outer causing a race (#12), sequential operations summing past the outer budget (#16). The escalation recovered correctly in all three cases, but the system-prompt addendum should surface the timeout hierarchy explicitly.

Detail — the three misalignments and their implication

#16time.sleep(60) + subprocess.run(..., timeout=10) sequential ≥ 70 s60 s Sleep alone consumed the budget; subprocess never started.

System-prompt addendum should state: inner timeouts must be strictly less than the python_repl outer timeout; for known-slow operations, raise the outer timeout with the timeout parameter on the call itself (max 600 s); sequential operations within one call must total less than the outer budget.

Unlike one-shot benchmark datasets, Omega offers: the user's daily work as a continuous real-world experiment; cargo mutants gate rigour on every commit (unusually precise signal); deep domain visibility to distinguish genuine improvement from benchmark over-fitting; and specific research angles (human-orchestrated subagents, seeded-handoff working method — documented in docs/subagents-research.html) that the literature has not studied in Omega's form.

🔬 Methodological note: the language-choice question

⚠ Retraction — earlier framing used cargo-culted language essentialism

Phrases like "orchestration is shell-shaped" or "Python is a natural fit for coding work" presuppose that languages have intrinsic domains they are for. This conflicts with Turing equivalence and conflates three empirically independent factors. The framing below is the correction.

The decomposed empirical framing

The actual phenomenon has three independent axes — none of which is "what language X is for":

  • Encoding cost c(L, P): the number of tokens needed to express operational pattern P in language L. Shell has very low encoding cost for pipe-composition; Python has low encoding cost for in-process data manipulation. Both are measurable.
  • Model fluency f(M, L, P): how reliably model M produces correct, idiomatic L-code for pattern P. A property of M's training distribution. Current frontier models have very high fluency in Python and Bash; much lower in Nushell or Steel.
  • Pattern density: how much of a benchmark's mass falls on pattern P versus other patterns.

Total cost is the sum over patterns of c(L, P) × density(P) / f(M, L, P). Lower is better. The language that minimises this sum for a given task and model is the better substrate — empirically, model-dependent, and contingent on the task distribution. No language-domain essentialism.

The Felleisen-expressiveness sharpening — why shell's apparent win is about encoding density, not expressive power

Shell's pipe-composition operator is macro-expansible into Python; the reverse is not true. Python is strictly more expressive in Felleisen's formal sense (1991): it can express patterns that shell cannot without a global program transformation. Shell's apparent win on orchestration-heavy tasks is therefore not about expressive power — it is about encoding density for chained subprocess invocation.

Ruby's backtick syntax (`cmd`) demonstrates the point: shell-level encoding density is achievable as syntax sugar inside a more expressive language. The sh() helper is the minimal instance of this for Python.

Why coding-agent work looks shell-aligned — without essentialism

Coding-agent work consists largely of stateful operations on a file tree via specialised external programs: compile, test, lint, format, grep, git, diff, etc. Those programs are CLI tools designed for shell. That is the load-bearing fact — contingent on Unix history, not on any intrinsic property of programming or shell. A future ecosystem with typed library APIs (pygit2, LSP clients, etc.) would shift the encoding-cost balance toward Python. The framing is stable; only the numbers would change.

⚠ Maintenance burden of structured tool wrappers

A library wrapping git, cargo, grep, etc. with typed APIs requires permanent maintenance: every subcommand, every flag, every output-format change ships a breakage. The LLM uses 5% of any such library; the other 95% rots. The sh() helper sidesteps this completely: it knows nothing about git, never breaks unless bash itself changes.

Forward-looking caveat

The analysis above reflects the current model generation's fluency distribution. A model trained on more diverse agentic-substrate traces could shift the picture substantially. The cost-fluency-density framing is robust to that shift: the same three axes apply, only the numbers change.

§3 RQ-A REPL alone

Status: F-B shows additive REPL is not measurable. F-C shows Tier 1 engages REPL on file-centric tasks. F-F shows full Tier 2 on a shell-centric task works end-to-end with genuine REPL state usage. The benchmark research phase for RQ-A is complete. The forward work is Phase 1–2: making the REPL user-grade, not running more benchmark confirmation trials. The broader Harbor sweep (87 remaining TB2 tasks) is deferred until the tool-selection mechanism (Phase 1.2) replaces the env-var flags in the Harbor invocation pattern.

RQ-A detail — value case, design leans, resolved questions

Value case: persistent REPL accumulates state (working directory, environment, loaded data, defined functions) across tool calls. CodeAct's +20% result is the strongest evidence — but the advantage is from action format, not persistent state. SRLM §3.5 warning applies at <131K tokens.

  • [fa3e44] How do existing tools integrate? → Currently additive tool plus flag-gated replacements; Phase 1.2 migrates to tool-selection list.

RQ-A questions

💡 [368041] What language is the REPL? RESOLVED

Python (subprocess-based, not PTY). High model fluency for subprocess, threading, and file-I/O patterns. Remaining encoding-cost gap on subprocess-heavy tasks addressed by sh() helper (Phase 1.3). PTY-based Bash REPL is Phase 3 work — a second independent implementation.

💡 [fa3e44] How do existing tools integrate with the REPL? RESOLVED → evolving in Phase 1.2

Currently: additive tool plus flag-gated replacements. Phase 1.2 replaces this with a tool-selection list at session creation. The "Tier" abstraction becomes a saved configuration in the UI (Phase 2.1).

💡 [c88a22] How does REPL state persist across process restarts? OPEN — deferred to Phase 2.3

strict_resume refuses REPL sessions loudly. The principled fix (replay python_repl calls into a fresh subprocess) is not implemented. See session-design.html §oq-repl-replay. For Phase 2, starting a new session is the resume story.

💡 [835f01] Cost / determinism / debuggability OPEN

Real costs observed: per-call timeout overhead, subprocess hangs, non-determinism in apt-get install timings. The python_repl_bootstrapped event and tee-forensics output make trajectories inspectable from events.jsonl. For llm_query recursion: not yet relevant; RQ-C not started.

💡 [c86aa6] Interaction with strict resume OPEN — deferred

Round-trip gate test not yet extended to verify REPL state equivalence. For benchmark use, acceptable (Harbor reruns). For daily-use REPL, the extension is a prerequisite. See session-design.html §oq-repl-replay.

💡 [b3e7f2] Does adding a sh() helper close the encoding-cost gap on shell-heavy tasks? RESOLVED → Phase 1.3

Folded into Phase 1 cleanup as an ergonomic improvement. Discriminating logic preserved: if cost drops and pass rate holds after adding sh(), encoding cost was the bottleneck. If behaviour doesn't shift, model fluency was the actual bottleneck. See § sh() helper.
