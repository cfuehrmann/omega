Vision / Screenshot Tool — Design Notes

Status: Not scheduled — design notes only · Extracted from docs/repl-and-substrates.html (formerly "Phase 1.5 — Vision tool") when the REPL experiment was closed. This capability is orthogonal to the REPL work and may be picked up on its own.

In one sentence

Give the agent eyes: a screenshot tool that captures the running Omega UI (or any URL) and returns the image to the model as a tool-result image content block, so the agent can visually verify what it built instead of relying solely on snapshot tests and the user as a visual oracle.

What

A screenshot tool that captures the running UI (or any URL) and returns the image to the model as a tool-result image content block. Anthropic's Messages API already accepts image content in tool results; the work is on Omega's side.

Why it would matter

The best ROI is validating UI work — the agent can verify that code blocks, truncation markers, tracebacks, and escalation events actually render as intended, instead of relying solely on snapshot tests plus the user as a visual oracle. It is also useful for catching CSS regressions and for any future UI work generally.

Cost / shape of the work

  • Schema-loud cross-crate change. Tool results currently carry text; they need to generalise to a sequence of content blocks (text + image), matching Anthropic's shape. Same shape of change as the tool-selection migration (former Phase 1.2).
  • Image persistence. Write PNGs to .omega/sessions/<id>/cache/images/<hash>.png and store a path-reference in events.jsonl — mirrors how cache/run/ works for shell tee'd output. Avoids bloating the event log with inline base64.
  • WS protocol + Leptos rendering. So the user sees what the agent saw, not just [image].
  • Capture mechanism. MVP: shell out to chromium --headless --screenshot (zero new Rust deps; works today). Richer interaction (click / type / scroll before snapping) needs a headless-Chromium Rust crate (chromiumoxide or headless_chrome) — defer until needed.

Open questions

  • Do we ship the MVP (one-shot screenshot) or invest in the interactive variant up front?
  • Should the tool be REPL-callable (so python_repl code can capture screenshots) or only top-level?
  • Token-cost budgeting — a 768×768 tile is ~250 tokens; full-page screenshots can be several thousand. Need a default viewport size.

Contract note

Generalising tool results from text to a content-block sequence is a schema-loud change to OmegaEvent (the canonical event type in crates/omega-types/src/events.rs). Per the contract-authority rule, the Rust types are the source of truth: change them first, then let persistence, the WS protocol, and the UI follow. The change should be visible in the on-disk format — old readers should fail loudly on the new syntax rather than silently misinterpret an image block as text.
