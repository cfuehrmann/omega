§1 Omega's multi-agent angles

§1.4 [a3f7c2] The human-orchestrated, artefact-returning angle

The multi-agent literature (Kim et al., Anthropic analysis, Cemri et al.) studies LLM-orchestrated subagents: parent emits tool call; child returns a summary string into parent's LLM context. That is the pattern the papers find wanting. Omega already uses a structurally distinct third pattern: the human orchestrates; the child returns artefacts on disk (commits, docs), not strings. No orchestration-token overhead, no Baseline Paradox, no context-pollution penalty. The planning project that produced this document was executed via this pattern. Omega's commit history is the evidence base. Nobody in the literature appears to have studied this regime rigorously — Omega is uniquely positioned to instrument and measure it.

§1.5 [e91b54] The seeded-handoff working method

A session runs until the LLM and user converge on handoff time (context drift, logical unit completion, or both). The LLM drafts a seed prompt; the user reviews and dispatches to a new or child session. Two flavours: successor handoff (session ends; child continues linearly) and sub-session handoff (session continues; child does a delimited task and returns artefacts). Because the orchestrator is human and returns are artefacts on disk, this pattern sidesteps the three failure modes the literature most consistently documents (context loss, error propagation, coordination overhead). It is incrementally automatable: trigger detection → seed-prompt drafting → dispatch, each step independently valuable. See §6 for the automation roadmap.

§2 Evidence base — anti-MAS findings

Evidence is presented from most to least negative for multi-agent coding setups. All findings concern coding / software-engineering tasks unless noted otherwise. The RLM papers' strongest results are on long-context retrieval tasks (6M–11M tokens) — a domain gap with Omega's typical sessions (<131K tokens). For REPL-specific evidence see repl-and-substrates.html §2.

Dependency: blocked on positive signal from RQ-A (REPL beneficial in isolation) and at least neutral from RQ-B (subagents not actively harmful). RLM is essentially free once both prerequisites exist (≈2 days given prereqs).

Lean: today's framing survives intact. "The session" is the outer Omega instance the user interacts with. Its REPL is a property of that instance. Children spawned by llm_query are child sessions in exactly the same sense as vanilla subagents. Not yet relevant; RQ-C not started.

§6 RQ-D Handoff automation

Status: Deliberately deferred while REPL benchmarks ran. The seeded-handoff working method (§1.5) continues to prove its daily value. RQ-D may rise in priority once Phase 1–2 complete, or independently as Omega's commit history accumulates enough data to study the pattern rigorously. No implementation work has started.

Unlike RQ-A/B/C, this builds on a pattern that has already proved its value in daily use. It sidesteps the three failure modes the literature consistently documents (context loss, error propagation, coordination overhead) by design — the orchestrator is human, and returns are artefacts on disk.

Incremental automation steps

  1. Trigger detection. The agent monitors context growth, turn count, and topic coherence; flags when a handoff is likely beneficial. Surfaces a suggestion to the user without acting. Instrument the current manual workflow in Omega's event log first: record each handoff event, the seed used, and the outcome (gate passage, commit quality, time elapsed).
  2. Seed-prompt drafting. When a handoff is flagged (or requested), the agent drafts the seed: current state, next goal, necessary context for the incoming session. User reviews and edits before dispatch.
  3. Dispatch. The agent initiates the child session with the reviewed seed, correct model, and effort. Records the SessionRef to the child in its event stream. Each step is independently useful without the next.

Three incremental steps (§6): trigger detection, seed-prompt drafting, dispatch. Each step is independently useful. The research question is: how much of the handoff does the agent need to automate before human review becomes the bottleneck rather than the safeguard? Empirical question; answer from instrumented use. No implementation work has started; sequenced after Phase 1–2.

💡 [a3f7c2] Is the human-orchestrated, artefact-returning pattern worth formalising as a research object? OPEN

The pattern is distinct from both vanilla MAS and REPL-based RLM, sidesteps known failure modes, and remains unstudied in the literature. Omega's session logs and commit history are a growing evidence base. Concrete research design: instrument the handoff events, measure outcomes, compare against sessions that did not use handoffs. Feasibility: high. Unique Omega advantage: high. Sequenced after Phase 1–2.
