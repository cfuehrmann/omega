Phase 3 RLM Exploration Proposal

Status: Proposal — for review · Resolves the open questions blocking Phase 3 in docs/session-design.html §rlm-decision · Written against codebase as of commit develop HEAD (May 2026) · Canonical path: docs/rlm-exploration-proposal.html

Introduction

Phase 2 (strict resume) is complete. Before Phase 3 can start, two strategic questions must be answered: whether to model Omega's multi-agent capability on Recursive Language Models (RLM), and in what order REPL and subagent primitives should be built. This document answers both, grounded in a survey of five reference agents, and resolves the remaining open questions from the design doc.

The RLM paper frames subagents not as tool calls that return strings but as first-class function calls inside a persistent REPL — the result lands as a Python variable, not as a string pasted into the LLM context. The survey below tests whether this distinction matters for Omega.

Ground rules (already decided — not revisited here)

✓ [b0bc06] Two independent runtime flags, four-cell matrix

Two flags — repl and subagents — each defaulting to off, independently switchable at runtime, recorded in SessionStartedEvent. The four combinations are:

subagents=off subagents=on
repl=off baseline (current Omega) vanilla subagents
repl=on REPL only (CodeAct-style) RLM (REPL + llm_query)

Config mechanism specifics are resolved in § [b0bc06].

✓ [abe7e7] Stay on develop; feature flags alone, no branch

All new code lands on develop behind flags. The gate runs every commit. No long-lived feature branch.

[b2e97e] Comparative survey

Five agents were surveyed across four dimensions: REPL availability, subagent support, invocation primitive, and parent–child data flow.

opencode — ~/opencode/

TypeScript/Bun. Multi-agent orchestration built on an effect-typed session loop. Subagents are defined as named agent configs with a mode field ("subagent" | "primary" | "all").

Key files inspected
  • packages/opencode/src/tool/task.ts — the TaskTool (v2.1.63 renamed to Agent tool): creates or resumes a child session, awaits its final text output, returns a string to the parent.
  • packages/opencode/src/session/prompt.ts — handleSubtask() drives the child's loop; the parent streams the child's tool-call events to the UI during execution, giving a "live" appearance, but still blocks until the child finishes and collects a single result string.
  • packages/opencode/src/agent/agent.ts — Info.mode field; subagent-type agents can be given their own model, tool allowlist, and permission rules.
  • REPL: No (shell access via Bash tool but no stateful interpreter namespace).
  • Subagents: Yes — via the Task/Agent tool_use.
  • Invocation primitive: tool_use (task / agent tool ID).
  • Data flow: Call-return. The child session produces a final text reply; that text is returned as the tool result string, landing in the parent's LLM context. Sessions can be resumed via task_id — stateful across calls but the boundary is still tool-call granularity.
  • Notable: Foreground (blocking) and background (concurrent, auto-deny-prompts) modes. Result string is wrapped in XML tags — <task_result>…</task_result>.

forgecode — ~/forgecode/

Rust. Clean separation of agent execution (AgentExecutor) from conversations. Each named agent becomes a tool definition at runtime.

Key files inspected
  • crates/forge_app/src/agent_executor.rs — the execute() method creates or reuses a ConversationId, calls ForgeApp::chat(), collects the final markdown reply via a streaming loop, and wraps it in ToolOutput::ai(Element::new("task_completed")…).
  • crates/forge_app/src/infra.rs — get_agents() / get_agent_infos() trait; agents are config-driven, not code-defined.
  • REPL: No.
  • Subagents: Yes — agent names surface as tool names.
  • Invocation primitive: tool_use (agent name ↔ tool name mapping).
  • Data flow: Call-return. The child conversation streams internally; the streaming output is collected and the final text is returned as a ToolOutput XML string in the parent's LLM context.
  • Notable: Can reuse an existing ConversationId (stateful resumption). Interrupts propagate — the parent's abort triggers AgentToolInterrupted.

pi-mono — ~/pi-mono/

TypeScript. A lean multi-package monorepo: packages/agent provides a generic agent loop; packages/coding-agent is the product built on top. Neither implements subagent orchestration at the library level. ~/pi/ (bare) does not exist.

Key files inspected
  • packages/agent/src/agent-loop.ts — standard LLM request → tool-call → result loop. No subagent spawning.
  • packages/agent/src/agent.ts — AgentOptions / AgentState; tool list is flat, no concept of agent-as-tool.
  • packages/web-ui/src/tools/javascript-repl.ts — a UI renderer for JavaScript code results (display only, not an agent REPL).
  • REPL: No (there is a JavaScript renderer in the UI but it is not an agent-level interpreter).
  • Subagents: No — single-agent loop; no orchestration primitives.
  • Invocation primitive: N/A.
  • Data flow: N/A.

fast-rlm — ~/fast-rlm/

Python/Deno. The reference RLM implementation. Each LLM call runs inside a Pyodide (WebAssembly Python) REPL sandboxed in a Deno process. Subagents are invoked via an await llm_query(prompt, schema) function that is pre-loaded into every REPL namespace.

Key files inspected + RLMConfig knobs

fast_rlm/_runner.py — the full Python driver:

  • run(query, config, tools, output_schema, env_variables) — entry point; serialises everything to temp files, shells out to deno run src/subagents.ts, reads back a JSON result.
  • Tools are Python callables; their source is extracted via inspect.getsource and executed inside Pyodide before the REPL starts.
  • Sub-agents do not inherit parent tools; must be explicitly forwarded via llm_query(prompt, schema, tools=[…]).

examples/: parallel_r_count.py shows six concurrent llm_query calls via asyncio.gather. http_smoke_test.py shows tool injection — a Python callable is extracted, sent to Deno, executed in Pyodide. Results land as Python objects (Pydantic-validated), not as strings in any LLM context.

  • REPL: Yes — Pyodide (WebAssembly Python) inside Deno. Sandboxed. HTTP routed through pyodide-http / Deno fetch.
  • Subagents: Yes — via llm_query() function in the REPL namespace.
  • Invocation primitive: Python function call in the REPL (not tool_use).
  • Data flow: Variables / Python objects. Results land as Python values (Pydantic-validated); they do not enter the parent's LLM context as text. The parent agent assigns results = await llm_query(…) and then works with the object programmatically.
  • Notable: Parallel spawning via asyncio.gather. Typed output schemas. Two runtimes (Python + Deno). Tools must be self-contained callables (no module-level closures). Sub-agents do not inherit tools.

Claude Code — web docs

Source not available locally; surveyed via official docs. Claude Code is Anthropic's first-party coding agent; its subagent model is the most elaborated of the five surveyed here.

Documentation highlights
  • Agent/Task tool: the parent invokes a named subagent by emitting an Agent (formerly Task) tool call. The subagent runs in its own context window and returns a result string.
  • Built-in subagents: Explore (read-only, Haiku), Plan (read-only), general-purpose. Subagents cannot spawn further subagents — anti-nesting guard is explicit.
  • Foreground / background: background subagents run concurrently; they auto-deny any tool that would otherwise prompt.
  • Session forking: a separate primitive (fork, not subagent) that branches the conversation tree; each fork runs independently and can be observed / steered.
  • REPL: code_execution_20250825 is a beta API-level tool for sandboxed Python execution (Anthropic side). An internal REPLTool exists for Anthropic engineers but is not exposed to users. No general-purpose user-facing REPL in the published agent.
  • Data flow: call-return. Result lands in the parent's LLM context as a summary string.
  • Memory: optional persistent memory directory (~/.claude/agent-memory/) per subagent, for cross-session learning (separate from the main context).
  • REPL: Partial — sandboxed code execution is an API-level beta tool, not a first-class agent primitive for end users.
  • Subagents: Yes — via Agent (formerly Task) tool_use.
  • Invocation primitive: tool_use (Agent tool).
  • Data flow: Call-return. Result string lands in parent's LLM context. Subagents are anti-nested (a subagent cannot spawn a sub-subagent).
  • Notable: Most mature subagent model. Session forking as a distinct primitive. Background agents (concurrent). Per-subagent model and tool config. No user-facing RLM-style REPL.

[cf2db0] REPL first vs. subagents first

✓ Recommendation — Subagents first

Implement vanilla subagents (the subagents=on, repl=off cell) before the REPL. The REPL follows in a subsequent iteration.

Rationale (full)

Arguments for subagents first

  1. Survey consensus. All three production coding agents surveyed — opencode, forgecode, Claude Code — ship subagents without a REPL. The RLM REPL-first approach is found only in the research implementation (fast-rlm). Omega is a coding agent, not a data-science notebook; the survey confirms that subagents alone are sufficient for a production-quality first deliverable.
  2. Immediate value. Vanilla subagents let the agent delegate tasks to child sessions today, using machinery (SessionId, EventId, SessionRef, Origin, strict resume) that Phase 0–2 already built. REPL adds capability later without invalidating subagents.
  3. REPL persistence is unsolved. The REPL introduces a live interpreter namespace (variables, function defs, imports) that does not naturally event-source. The three candidate strategies (replay, snapshot, hybrid — see § [c88a22]) all have significant tradeoffs. Building subagents first defers this complexity to a later, focused iteration.
  4. Two runtimes vs. one. The Pyodide-in-Deno approach (the only battle-tested RLM sandbox we surveyed) adds a Deno process and a WASM Python interpreter to a Rust codebase. Subagents add nothing — they re-use the existing Omega process and event store.
  5. run_command already covers the coding-agent REPL use case. Omega's existing run_command tool gives the LLM arbitrary shell execution. The primary advantage of a REPL over repeated shell calls is statefulness (accumulated variables, loaded data). For code editing this advantage is real but marginal; for data manipulation (pandas, numpy) it is much larger — but that is not Omega's primary use case.
  6. Subagents are REPL prerequisites, not vice versa. RLM requires that each llm_query call maps to a child session ([8e161a]). The child session requires SessionId, EventId, SessionRef, and Origin — all of which Phase 1–2 already provide. Building subagents first means we exercise this infrastructure before adding the REPL layer. The RLM cell (repl=on, subagents=on) can then add the REPL-level orchestration on top of working subagents.

Arguments for REPL first (and why they lose)

  • "REPL is the substrate RLM is built on." True in the paper's framing; false for Omega in the immediate term. Omega does not need RLM to be useful; vanilla subagents are useful on their own and are a stepping stone toward RLM, not away from it.
  • "Vanilla subagents lock in the wrong parent–child data-flow." Only partially true. The result-as-string model is simpler and is what every production agent uses. The flag matrix means both can coexist; vanilla subagents do not prevent the REPL-variable model from being added later in the repl=on cells.

[368041] REPL language

✓ Recommendation — Persistent shell session (PTY-based)

When the REPL is eventually built, use a persistent shell session driven by a PTY rather than embedding a Python interpreter. If data-science workloads become important later, add Python as a second REPL kind behind its own sub-flag.

Language options and tradeoffs
Option Pros Cons Verdict
Shell (PTY) Zero new runtime dependencies; bash is universal; every command already works; deterministic enough for replay; matches Omega's coding-agent persona; familiar to the LLM Output parsing is harder than structured returns; PTY boundary detection is fiddly; non-interactive commands still need careful stdout/stderr demuxing ✅ Recommended
Python via Pyodide-in-Deno Battle-tested in fast-rlm; WASM sandbox; full Python ecosystem (numpy, pandas) Two foreign runtimes (Deno + WASM); large binary; non-deterministic (time, network); hard to replay ❌ Too much runtime complexity for Omega's use case
Python via PyO3 Single foreign runtime; Rust-native bridge; reasonably fast startup Sandboxing is harder; Python GIL; import side-effects; non-deterministic for replay ⚠ Viable but deferred; re-evaluate when data-science workloads appear
Rust-native scripting (rhai / rune) No foreign runtime; embedded cleanly; deterministic Loses the Python ecosystem; LLM training data biased toward Python/shell; syntax unfamiliar to most ❌ Too niche; LLM proficiency too low

Shell fits Omega's purpose. Omega is a coding agent that edits files, runs builds, and executes tests. All of these are shell operations. A persistent PTY session accumulates working-directory state, environment variables, and shell functions — exactly the state that matters for code work. Unlike Python REPL state (dataframes, intermediate results), shell state is already implicitly managed by our existing tools; the PTY just makes it stateful across individual tool calls.

Replay note. Shell commands are more deterministic than Python REPL cells: a git checkout main produces the same result on replay; a random.random() call does not. This makes replay-based persistence (see § [c88a22]) significantly more reliable for shell than for Python.

[fa3e44] REPL vs. tool_use integration shape

✓ Recommendation — REPL as another tool, not a replacement

The REPL is exposed as a new tool alongside the existing tool set. It does not replace tool_use. The LLM chooses the REPL when it benefits from accumulated state; it continues using individual tools (read_file, run_command, etc.) for one-shot operations.

Full rationale and integration sketch

The two extremes from the design doc:

  • (a) REPL replaces tool_use entirely. Every capability is exposed as a function in the REPL namespace. This is the RLM article's design. It is a clean architecture but a large, risky migration.
  • (b) REPL is another tool. The existing tool set remains. A repl_exec (or shell_session) tool is added that feeds a command into the persistent session and returns the output. The LLM can use REPL or individual tools as appropriate.

Option (b) is recommended because:

  1. Preserves existing tool machinery. The full test suite, the tool-call event log, and all existing tool implementations remain valid. Switching to (a) would require re-implementing every tool as a Python / shell function in the REPL namespace.
  2. Matches the flag semantics. repl=on adds capability; it does not rewrite the agent. The LLM's tool list grows by one entry; everything else stays the same.
  3. All three production agents use (b)-equivalent designs. opencode, forgecode, and Claude Code add capabilities as tools; they do not replace their tool sets with a REPL.
  4. Progressive upgrade path. Under (b), moving toward the RLM llm_query-in-REPL model later is a matter of adding llm_query as another pre-loaded REPL function — it does not require tearing out the tool layer.

Sketch of the new tool (when REPL is built): 

// Tool definition (in ToolDefinition list, only when repl flag is on)
{
  "name": "shell_session",
  "description": "Execute a shell command in the persistent session REPL. State (working directory, env vars, shell functions) persists across calls.",
  "input_schema": {
    "type": "object",
    "properties": {
      "command": { "type": "string", "description": "Shell command to execute" }
    },
    "required": ["command"]
  }
}

// Corresponding event (recorded in events.jsonl)
{
  "type": "repl_executed",
  "event_id": "...",
  "command": "cargo test",
  "stdout": "...",
  "stderr": "...",
  "exit_code": 0
}

When repl=on, subagents=on (the RLM cell), llm_query is added as a second pre-loaded function: the REPL namespace then has both shell execution and child-agent invocation, and the orchestration layer switches from tool_use to REPL-function calls.

[c88a22] REPL state persistence sketch

✓ Recommendation — Event-replay with determinism pragmatics

Each REPL command is recorded as a ReplExecuted event. On resume, commands are replayed in order to rebuild the session state. Non-deterministic side effects are accepted as a known limitation; failures during replay surface an error and offer a clean-start escape.

Full rationale, strategies compared, Rust sketch

Strategy comparison

Strategy How it works Pros Cons
Replay Every executed command stored as an event. Resume re-executes in sequence. Pure event-sourcing; fits existing fold(events) model; no binary blobs Non-determinism: file I/O, time, randomness, network calls may differ on replay
Snapshot Interpreter state serialised (e.g. PTY env dump) into an event payload. Fast resume; exact state recovery Binary blobs in events.jsonl violate Contract Authority; PTY state is not easily serialisable
Hybrid Events record commands; snapshots cached out-of-band keyed by event ID; resume uses latest snapshot + forward replay. Balances accuracy and speed Two storage layers; cache invalidation; more moving parts

Why replay wins for a shell REPL

Shell commands are far more deterministic than Python REPL cells. A cd /tmp && ls produces the same output on replay. A git status reflects actual repository state. Most failure modes (file not found, permission denied) are exactly what we want to surface on resume. The remaining non-determinism (wall clock, network I/O) is tolerable for a coding agent where the primary state is the filesystem, not in-memory objects.

Snapshots violate the Contract Authority rule: binary blobs in events.jsonl would make the schema opaque and hard to evolve. The hybrid approach adds complexity that is not justified at MVP.

Rust event type sketch

/// Recorded when the REPL executes a command.
/// On resume: commands are replayed in order to rebuild session state.
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(tag = "type", rename = "repl_executed")]
pub struct ReplExecuted {
    pub command: String,
    /// Captured stdout (may be truncated for large outputs).
    pub stdout: String,
    /// Captured stderr.
    pub stderr: String,
    pub exit_code: i32,
}

/// Recorded when the REPL session is reset (e.g. replay failure on resume).
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(tag = "type", rename = "repl_reset")]
pub struct ReplReset {
    pub reason: ReplResetReason,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum ReplResetReason {
    /// Replay failed with this error; user accepted a clean start.
    ReplayFailed { error: String },
    /// Explicit user request.
    UserRequested,
}

Resume algorithm

  1. Fold events up to the last resumable boundary.
  2. Collect all ReplExecuted events since the last ReplReset (or the beginning, if no reset).
  3. Spawn a fresh PTY session; replay commands in order.
  4. If any command exits non-zero or produces unexpected output, log the divergence. Continue replay (the agent can observe and correct on the next turn) unless a command causes an unrecoverable error (e.g. PTY dies), in which case emit ReplReset and offer a clean start.

Interaction with the round-trip gate test ([c86aa6]). When REPL is built, the gate test must verify that REPL state is equivalent after resume. For a shell REPL, "equivalent" means the working directory and key environment variables match; it does not require byte-for-byte stdout identity.

[b0bc06] Flag config mechanism specifics

✓ Recommendation — Env vars only (for now); typed struct in SessionStartedEvent

Resolve flags from env vars at startup. Record the resolved combination in a typed features struct on SessionStartedEvent. Do not expose via config file at this stage.

Env var names

OMEGA_FEATURE_REPL=1        # enable the REPL tool (default: unset / 0 = off)
OMEGA_FEATURE_SUBAGENTS=1   # enable subagent spawning (default: unset / 0 = off)

Any non-empty, non-zero value is treated as true. Unset or 0 is false. This matches Harbor's environment-sweep model: four benchmark runs, each setting a different combination, without rebuilding Omega.

Why not a config file?

Config file rationale (collapsed)
  • Env vars are sufficient for Harbor sweeps; a config file adds a parsing layer without improving expressiveness for boolean flags.
  • Config files create a second source of truth that can conflict with env vars (precedence questions). Env vars have a well-understood override model (child process inherits; can be overridden per-run).
  • Config file exposure is worth adding if the flags grow in number or gain non-boolean values (e.g. REPL interpreter choice, subagent depth limit). At that point a TOML section in omega.toml or a dedicated [features] block is appropriate. Defer until the need is concrete.

SessionStartedEvent — features field

The resolved flag combination is recorded at session creation time, not re-read on resume. This ensures post-mortems and benchmark logs can determine exactly which combination ran. 

/// Typed features struct — add new fields here as new flags are introduced.
/// Serialises to/from JSON as camelCase keys.
#[derive(Debug, Clone, Default, Serialize, Deserialize, PartialEq)]
#[serde(rename_all = "camelCase")]
pub struct SessionFeatures {
    /// Whether the REPL tool is enabled for this session.
    pub repl: bool,
    /// Whether subagent spawning is enabled for this session.
    pub subagents: bool,
}

// In SessionStartedEvent:
pub struct SessionStartedEvent {
    pub session_id: SessionId,
    pub omega_commit: String,
    pub model: String,
    pub effort: String,
    pub origin: Origin,
    /// Resolved feature flags at session creation time.
    /// Both default to false (preserving current behaviour).
    pub features: SessionFeatures,
}
JSON shape and backward-compat note

On the wire (events.jsonl):

{
  "type": "session_started",
  "sessionId": "...",
  "omegaCommit": "abc1234",
  "model": "claude-sonnet-4-6",
  "effort": "medium",
  "origin": { "type": "root" },
  "features": {
    "repl": false,
    "subagents": false
  }
}

Schema evolution note. Per the Contract Authority rule, old sessions without a features field should not be silently defaulted (that would mask schema drift). Sessions produced before this field existed will fail to deserialise into the new SessionStartedEvent — that is the intended loud failure. If a best-effort migration path is later desired, it must be deliberate (a migration script, not a #[serde(default)] band-aid).

Exception: if the team decides that defaulting features to {repl: false, subagents: false} for old sessions is semantically correct (old sessions ran without either feature, so the absent field really does mean both-off), then #[serde(default)] on features alone is justified. This is the only case where a default is semantically meaningful, not defensive.

Startup resolution pseudocode

fn resolve_session_features() -> SessionFeatures {
    SessionFeatures {
        repl:      env_flag("OMEGA_FEATURE_REPL"),
        subagents: env_flag("OMEGA_FEATURE_SUBAGENTS"),
    }
}

fn env_flag(name: &str) -> bool {
    std::env::var(name)
        .map(|v| !v.is_empty() && v != "0")
        .unwrap_or(false)
}

The resolved SessionFeatures is passed into the agent constructor and stored in SessionStartedEvent. From that point forward, all flag-gated code paths read from the event, not from the environment — this is important for determinism during strict resume.

Briefer leans — remaining open questions

The following questions are not yet resolved in full; each gets a directional lean and notes on dependencies.

[8e161a] How does an llm_query call relate to SessionId?

Lean: each llm_query invocation is its own session. Mirrors the existing decision that subagents are child sessions (Origin::SubagentOf). When the REPL is built and the RLM cell is active, the REPL function llm_query(prompt) is syntactic sugar over the same spawn-a-child-session machinery used by vanilla subagents; only the invocation surface (REPL function vs. tool_use) differs. The child gets its own SessionId, events.jsonl, and folder. The parent's ReplExecuted event records the returned value (or a reference to the child SessionId if the value is large).

[053c4a] What is "the session" in an RLM Omega?

Lean: today's framing survives intact. "The session" is the outer Omega instance (the one the user interacts with). Its REPL is a property of that instance, not a separate session. Child sessions spawned by llm_query are child sessions in exactly the same sense as vanilla subagents. The tree of sessions (parent + children + grandchildren up to max_depth) is visible in the event-sourced session graph, not a new concept.

[549288] Safety / sandboxing

Lean: inherit Omega's existing model for the shell REPL; add explicit sandboxing boundary if/when Python is adopted. Today's run_command has no sandbox beyond OS permissions — a persistent shell REPL raises the same risk by degree (accumulated state), not by kind. The existing permission / approval model is sufficient for a first iteration. If Python (PyO3 or Pyodide) is added later, a sandboxing story (seccomp, WASM boundary, or at minimum network and filesystem restrictions) must be designed before exposing it. The RLM article's Pyodide-in-Deno approach is the reference: Deno's --allow-* flags provide a coarse-grained sandbox.

[bab784] User interaction with an RLM Omega

Lean: chat messages stay; REPL is internal; UI optionally exposes REPL state for inspection. The user continues to send prose messages and receive prose replies. The REPL is an implementation detail of the agent loop, visible in the event feed (each ReplExecuted event is rendered as a collapsible code block with stdout/stderr) but not a first-class UI primitive. A dedicated "REPL console" pane is a Phase 4 feature, not a prerequisite.

[1f28f7] Output format — autoregressive vs. variable

Lean: autoregressive prose at the user boundary; variable returns only within the REPL and in subagent-to-parent data flow. FINAL(variable)-style returns (from fast-rlm) make sense when the consumer is another program (a parent REPL). They do not fit Omega's chat-style UI where the user expects a readable response. For the outer agent, the reply is always prose. For inner subagents returning to a parent REPL, structured returns (Pydantic-validated objects in the Python case; JSON in the shell case) are appropriate and will be designed in the REPL iteration.

[c86aa6] Interaction with strict resume

Lean: the round-trip gate test must be extended to cover REPL state when REPL is built. Today the gate asserts equivalence of active_model, active_effort, history, and context_hashes. When the REPL becomes part of Agent state, the gate must additionally assert that the replayed shell session ends in an equivalent working directory and environment. The definition of "equivalent" is deliberately loose (not byte-for-byte stdout) — see the persistence section above. This extension is a prerequisite for the REPL iteration, not for the vanilla-subagents iteration.

[835f01] Cost / determinism / debuggability

Lean: real costs, worth tracking explicitly. A persistent REPL adds latency (PTY startup, per-command round-trip), non-determinism (especially if Python is used), and debugging complexity (an LLM-driven shell trajectory has more failure modes than discrete tool calls). None of these kill the proposal, but all should be weighed against the benefit. For the shell REPL specifically: startup cost is small (bash is fast); commands are deterministic enough; debugging is aided by the ReplExecuted event log (every command and its output is recorded and visible in the feed). The cost of RLM-style llm_query recursion (additional LLM calls, token cost) is captured by the max_depth and max_money_spent analogues from RLMConfig, which Omega will need to implement when the RLM cell becomes active.
