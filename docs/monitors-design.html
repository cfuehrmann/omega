1. Goal & motivation

Omega today has wait_for_output, which is synchronous: it occupies the single tool-call slot and blocks the agent loop for the whole wait. We want an asynchronous monitor primitive instead — the agent registers a condition and keeps working (or parks), and when the condition fires, a message is injected as a user turn at the next valid seam (turn boundary). This is the Claude Code "Monitor" model. No preemption — monitors never interrupt an in-flight turn; they wait for the next boundary.

Target use cases (all expressible as one shell-command primitive):

  • Wait a duration — sleep N; echo done.
  • Wait for a process to finish — wait $pid; echo done.
  • Arbitrary boolean condition — a poll-and-if loop that prints only when the condition flips.

3. The seam rules (Anthropic Messages API — correctness, not opinion)

  • Two roles only: user / assistant. Anything not generated by the model — tool results, hook output, monitor notifications, human input — must enter as role:user. "Impersonating the user" is forced by the protocol, not a design choice; both Claude and opencode do it. The synthetic flag is only a harness/UI marker to distinguish it from a human-typed turn.
  • tool_use → tool_result is rigid: every tool_use block must be immediately followed, in the very next message, by matching tool_result blocks. You cannot insert a separate message between them.
  • Exactly two legal injection points for a user message:
    • Seam A — after the assistant ends with end_turn and is idle → start a fresh user turn.
    • Seam B — piggybacked after the tool_result blocks in the user message that answers a tool_use.
  • A "turn boundary" = end_turn OR a completed tool_result. "Generating" and "ToolRunning" are merely between boundaries. A generation always resolves into one of the two boundary kinds; if it resolves to a tool_use, you reach Seam B once the tool returns.
  • No preemption at the context level. The only way to act before the next natural boundary is to abort the in-flight request (transport level) — that "manufactures a seam now." We keep abort as a separate, explicit interrupt path; it is not how monitors deliver.
  • A long/blocking tool call holds Seam B shut for its whole duration → starvation. This is precisely the wait_for_output problem, and the reason monitors must be an out-of-band channel rather than a blocking tool.

4. Core mechanic: pull-at-boundary (single writer)

  • Monitors run as background OS processes. Each stdout line is enqueued to an in-memory queue. Monitors never write events.jsonl.
  • The main loop is the single writer. It drains the queue only at the two boundaries:
    1. right after it appends a tool_result, before calling the model again; and
    2. when it is about to park at end_turn.
  • When parked, the loop blocks on a select over { human input, monitor queue, roster-changed }. Any of them wakes it; it drains the queue and runs.
  • No-preemption is free: the loop simply never inspects the queue mid-turn.
  • The monitor subsystem is dumb — it knows nothing about the fold or the tool_use/tool_result invariant. That ignorance is the safety; only the loop, which is always at a legal boundary when it drains, ever appends.

5. Guardrail philosophy

Explorative minimalism. Add only guardrails that are obviously load-bearing for correctness. Deliberately skip the rails Anthropic added to protect a broad public. Iterate on agent features by observing the single agent's behavior, not by pre-empting every footgun. Omega's Web UI makes this safe: human-in-the-loop observability substitutes for automated kill.

KEEP — obviously load-bearing (correctness)

  1. Never inject between tool_use and its tool_result — enforced structurally by pull-at-boundary (no decision to get wrong).
  2. Single writer — only the main loop appends events; monitors enqueue.
  3. Distinct event variant for monitor injections (not a reused human user-turn) — per the AGENTS.md loud-schema-change rule. Projects to role:user.
  4. Identity + lifecycle + stop — each monitor has an id; start/fired/stopped attributable; agent can stop by id; stopping a dead monitor is a no-op.
  5. Process-tree reaping on stop and on session end (no orphans).
  6. Defined park/terminate condition — park iff a live monitor or queued events exist; otherwise terminate. Never hang when nothing can ever fire.

DROP — public-serving rails (skip now, re-add on evidence)

RailWhy we skip it
Default + hard-max timeout_msProtects strangers from forgotten monitors. We have explicit stop + session-end reaping. No timeout default; observe whether the agent leaks monitors first.
Auto-stop chatty monitors (firehose)Public-protective. We are one human watching one agent. UI count + activity is the backstop. Observe whether the agent self-corrects.
"Always use grep --line-buffered" / poll-interval naggingPrompt nannying. Ship the raw primitive; add guidance only if observation shows laggy/buggy monitors.

Tool surface

Ship the raw, general primitive first: monitor(description, command) — stdout lines are events; returns immediately with a monitor id. Plus stop_monitor(id). No sugar (monitor_until/_for/_process), no timeout, no auto-stop, no baked-in filter advice. Add a typed helper only if observation shows the agent repeatedly fumbles the shell boilerplate.

6. Signed-off decisions

#Decision
1No auto-stop. Runaway monitors are caught by visibility, not automation. Visible in both logs and UI. The UI is what earns the right to skip auto-stop.
2Batch at seam. Everything pending is delivered together at the next boundary. Canonical events stay distinct (per source); the context projection merges consecutive user-role content into one API message.
3stderr → events.jsonl (forensics paramount), but excluded from the context projection. New event class: diagnostic / non-projected — in the canonical log + UI, never in the model context (no token cost, never wakes the agent). Accept a chatty log; coalesce later; no stable event format needed yet.
4Unify the pending queue. Monitor lines and human input flow through one queue. Guiding principle: "when I (the human) queue a message, I am a human monitor." Items tagged by source (human / monitor-id). Default = enqueue (delivered at next seam, no preempt). Explicit interrupt (abort) stays a separate path (Enter = enqueue; explicit interrupt = abort/preempt the turn).
5Visualize the pending queue in the UI, including human messages.
6Unexpected/killed terminations surface into context (agent must learn so it does not wait forever). Clean agent-initiated stops need not.
7Collapsed-badge first enhancement = events-fired/activity (the runaway detector), ahead of timing stats.
8Monitor liveness is runtime-only, never reconstructed from the log. But forensics is fully in the log.
9Clean shutdown kills all monitors + best-effort writes MonitorStopped. Hard crash leaves a dangling MonitorStarted — that is intended forensic signal, not a bug.
10Resume forks a new session (see code facts); no live-monitor inheritance; nothing to reconcile.
11Headless = benchmark-only, bounded by an external time limit + session-end cleanup. Residual token-burn risk from a runaway monitor = observe, don't guard.

7. Layer mapping (Contract Authority — types are the schema)

Contract rule (from AGENTS.md): the Rust OmegaEvent in crates/omega-types/src/events.rs is the canonical schema. Persistence, WS protocol, and UI are derived projections. A semantic change must get a syntactic change (new variant), not a silent alias. Update the UI to match the types — never the reverse.
LayerWhat lives here
Canonical OmegaEvent MonitorStarted, MonitorDelivery (the batched synthetic turn, projects role:user), MonitorStderr (diagnostic, non-projected), MonitorStopped. See §8.
Persistence events.jsonl = all events. context.jsonl = derived projection — it excludes MonitorStderr and merges consecutive user-role events into one API message. (events.jsonl ⊋ context.jsonl.)
WsMessage (ephemeral; may carry extra fields) Live roster (per-monitor status, start time, runtime, event-fired count, stderr tail); pending-queue contents tagged by source; UI→server KillMonitor(id) control frame.
UI Collapsed badge (active count + activity) → modal (roster); pending-queue visualization; later: per-monitor detail + kill button.

8. Proposed event schema (sketch — finalize in events.rs first)

Field names illustrative; the point is the variant boundaries:

// canonical OmegaEvent variants (camelCase on the wire)
MonitorStarted { id, description, command, time }
MonitorDelivery {                       // drained at a seam; projects role:user
    time,
    items: [ { source, lines } ]        // source = Human | Monitor(id); per-source attribution
}                                        // (human turns MAY stay their existing user-message
                                         //  event, drained alongside — see open item)
MonitorStderr  { id, chunk, time }       // DIAGNOSTIC: in events.jsonl, NOT in context.jsonl
MonitorStopped { id, reason, exit_code, time }
                                         // reason = AgentStopped | UserKilled | ProcessExited | Crashed
  • The context fold must skip MonitorStderr.
  • MonitorDelivery (and any human user-turn drained with it) projects to role:user; consecutive user-role events merge into one API message at the projection boundary (satisfies role alternation + "batch at seam").
  • Unexpected MonitorStopped (UserKilled/Crashed/ ProcessExited while the agent depends on it) should be projected into context so the agent learns; clean AgentStopped need not.

9. UI design

  • Collapsed badge — shows the active-monitor count. First enhancement: events-fired / recent activity (so a single spewing monitor is visible at a glance). This is the runaway backstop.
  • Modal — roster of active monitors. Later: per-monitor start time, run time, events fired, stderr tail.
  • Pending-queue visualization — what the agent will see at the next seam, each item tagged human / monitor-id.
  • Kill button (longer-term) — per-monitor kill. Killing = terminate process tree immediately (server-side, urgent) + enqueue a MonitorStopped(UserKilled) notice drained at the next seam (no preempt). If the agent was parked on the last live monitor, the kill must wake the park (roster-changed wake reason) so the loop re-evaluates/terminates rather than hangs.

10. Lifecycle, shutdown, resume, headless

  • Within a running session: monitors are live runtime entities in the in-memory roster.
  • Clean shutdown: kill all monitor process trees + best-effort write MonitorStopped for forensics.
  • Hard crash / kill -9: dangling MonitorStarted with no Stopped; a forensic reader infers "process died here." Acceptable — intended signal.
  • Resume (handle_resume_session): forks a new session from a resumption basis; does not inherit live monitors. Nothing to reconcile. Verify nothing tries to revive monitors from the prior log.
  • WS reconnect/replay (should_replay): re-sends logged events to render the view; live roster reflects whatever the running process currently holds (empty after a restart). Liveness is never reconstructed.
  • Headless: benchmark-only, bounded by an external time limit + session-end cleanup. No human-kill needed there.

11. Implementation steps (phased)

Order is contract-first: schema before runtime before UI. Run targeted mutation testing (cargo mutants -p <crate> --cap-lints=true --file <path>) after each non-trivial logic chunk; add a Justfile recipe per run (template: mutants-system-prompt-guard). Tests end-to-end: omega-tools via execute_tool, omega-agent via Agent::send_message + MockProvider.
  1. Phase 0 — Schema. Add the four OmegaEvent variants in crates/omega-types/src/events.rs. Mark MonitorStderr non-projected in the context fold. Ensure the projection merges consecutive user-role events into one API message. Tests: fold invariants, projection merge, non-projection of stderr.
  2. Phase 1 — Monitor runtime. Process manager: spawn shell command, stdout line → queue, stderr → (chatty) MonitorStderr events, process-tree track + kill. Ephemeral roster (id, description, command, status, start time, event-fired count, stderr tail). Tools monitor() / stop_monitor() (returns immediately).
  3. Phase 2 — Loop integration. Unified pending queue (monitor lines + human input). Drain at the two seams. Parking select over {human, queue, roster-changed}. Terminate condition. Route human input through the queue; keep explicit abort/interrupt as a separate path.
  4. Phase 3 — WS + UI. Push ephemeral roster + pending-queue over WS; KillMonitor control frame. Badge + modal + queue visualization. Later: per-monitor detail + kill button.
  5. Phase 4 — Shutdown / resume / headless. Clean shutdown kills + logs MonitorStopped. Verify resume does not revive monitors. Headless relies on session end + external time limit.

12. Open items (deferred, non-blocking)

  • stderr coalescing granularity — by-lines vs by-time. Decided: accept chatty log for now, optimize later; no stable format required.
  • Human-in-delivery representation — does MonitorDelivery aggregate human + monitor items in one event, or does the human turn stay its existing user-message event drained alongside a monitor delivery? Lean: keep them as separate canonical events; the projection merges them. (Either way the UI tags by source.)
  • Exact reason enum for MonitorStopped and which reasons project into context (lean: all "unexpected" ones do).
  • Does MonitorStarted re-surface into context? Lean: no — the tool result already informs the agent; the event is for log/causality.
