Async Monitors — Invariants, Decisions & Future Work

CORE SHIPPED This doc keeps only what's needed to move forward: architectural invariants (must-holds), locked-in decisions (must-not-revert), the future menu, and a resume map. The history of how each piece was built lives in the code and its tests.

What exists

An agent can monitor(description, command) to run a background shell command whose stdout lines are injected as role:user at the next seam (no preemption), watch/steer it from the Web UI, and stop_monitor(id). Input (human + monitor) flows through one queue; controls are Queue / Halt / Abort. A first-class event sink lets out-of-band events (stderr, control actions) reach events.jsonl + UI when they happen.

1. Invariants & locked-in decisions

Seam & loop correctness (forced by the Anthropic Messages API)

  • Two roles only. Anything not generated by the model (tool results, monitor output, human input) enters as role:user. The synthetic flag is a harness/UI marker only.
  • tool_use → tool_result is rigid — the result is the very next message; nothing inserts between them.
  • Two legal injection seams. Seam A: after end_turn (idle). Seam B: after the tool_result blocks answering a tool_use. The loop drains the input queue only at these two points — so “never inject mid-pair” is structural, not a decision that can be gotten wrong.
  • No context-level preemption. Monitors never interrupt an in-flight message. The only pre-boundary action is Abort (transport-level), kept as a separate explicit interrupt.
  • ⛔ Monitors are out-of-band, never a blocking tool. A long/blocking tool holds Seam B shut → starvation (the wait_for_output problem this feature exists to fix).
  • Single writer for context-bound events — the loop appends; monitors enqueue. (Generalised by the event sink for non-context events — see below.)
  • Park/terminate is defined. Park iff a live monitor or queued item exists, else terminate; never hang when nothing can fire. Parked, the loop selects over {queue item, roster-changed}; roster-changed fires when the live set shrinks without a drainable item (e.g. a UI kill while parked).
  • Process-tree reaping on stop() and session end (process-group kill); no orphans.

Contract Authority & schema

  • The Rust OmegaEvent is the canonical schema (crates/omega-types/src/events.rs). Persistence, WS, and UI are derived projections. A semantic change requires a syntactic one (new variant), never a silent alias; update the UI to the types, never the reverse. Old logs failing to deserialize a renamed variant is intended signal, not a bug to paper over.
  • ⛔ Derived projections must be derived, not hand-mirrored. Frontend event frames deserialize straight into OmegaEvent; a drift-guard test round-trips every variant; an unparseable frame surfaces visibly. (Re-introducing a parallel hand-kept WsMessage enum is exactly the drift that made monitor deliveries silently invisible.) Genuinely transport-only envelope frames are the only hand-written part; a new one must not collide tags with an event.
  • Persistence shape. events.jsonl = all events; context.jsonl = derived per-record projection (keyed by ContextHash) that excludes MonitorStderr. Merging consecutive user-role records into one API message is request-assembly (project_messages()), leaving the per-record log un-merged so ContextHash foreign keys stay stable.
  • Projection rules. The fold skips MonitorStderr; MonitorDelivery projects role:user; an unexpected MonitorStopped (StoppedByUser / ProcessExited / ProcessCrashed) projects so the agent learns, while StoppedByAgent and StoppedBySessionEnd do not.
  • Roster is ephemeral — runtime-only, never persisted or reconstructed. WsMessage::MonitorRoster is transport-only (not an OmegaEvent), pushed seam-aligned on connect/reset/resume and after each monitor event.

Unified input model

  • ⛔ One persistent run loop + one InputQueue for human and monitor input, tagged by source. (Do not revert to parking inside send_message: it held the agent lock for the whole turn and deadlocked a fresh UserMessage.)
  • role:user is a projection, not a storage decision. Human messages stay UserMessage, monitor deliveries stay MonitorDelivery; merged late by project_messages(). Human/monitor symmetry falls out for free.
  • Three orthogonal controls: Queue (always enqueue, deliver at next seam), Halt (stop at next seam and wait), Abort (cancel the in-flight block). The retired pause/continue-for-injection machinery is guarded by a source-scan test so it can't grow back.

Event sink — event-time vs commit-time

  • ⛔ Event-time and commit-time are independent; keep both. Event-time = embedded time / UUIDv7 event_id (when it happened). Commit-time = file/arrival order in events.jsonl (when recorded). Never sort the log by time; the UI shows arrival order with each row's own time. Their divergence is signal (skew/latency/races), not noise.
  • One occurrence = one creation = one time = one event_id, routed to both disk and wire from that single value. Stamp time at the moment of occurrence; append whenever. (Re-deriving a fresh now at broadcast is the double-stamp bug.)
  • Sink contract: emit(event) appends + broadcasts in arrival order, from any caller at any time, with no coordination — EventStore is stateless and its append is O_APPEND/per-line atomic; ws_tx is a multi-producer sender resolved live (replaced on reconnect).
  • User action vs effect. A model/effort change is recorded at action-time (out-of-band, even mid-turn); its effect is turn-aligned — drive_turn snapshots model+effort at entry, so a mid-turn change applies next turn, never to the in-flight one.
  • Phase A (additive sink alongside the loop's path) is shipped; Phase B (sink as the sole emission path) is future work.

Guardrail stance & teaching copy

  • Explorative minimalism. Add only correctness rails; skip the public-serving rails Anthropic adds for a broad audience; iterate by observing the single agent. UI observability is what earns skipping automated kill.
  • Currently dropped (re-add only on evidence): default/max timeout_ms, auto-stop of chatty monitors, line-buffering nagging. Verified prior art (Claude Code) keeps auto-stop + line-buffering — flagged in Future work.
  • No auto-stop — runaway monitors are caught by visibility (logs + UI fired-count), not automation.
  • Teaching copy is load-bearing for usability. Tool descriptions + system-prompt addendum must say: monitor output is not the human; do not fabricate/guess output; after starting, end your turn and wait; stop when no longer needed. The tool result stays neutral bare data (a “stop it when done” sentence tipped Sonnet into killing monitors instantly).
  • Lifecycle decisions. Clean shutdown kills all trees + best-effort MonitorStopped; a hard crash leaves a dangling MonitorStarted (intended forensic signal). Resume forks a new session — no live-monitor inheritance, nothing to reconcile. Headless is benchmark-only, bounded externally + session-end cleanup.

2. Future work (sorted by attractivity)

Ranked by (value × readiness) ÷ (cost × risk). Nothing is scheduled — this is the menu.

  1. Kill button / KillMonitor control frame. Completes the UI control triad (today you can start + observe but not stop from the UI). Low-risk, well specified: a MonitorManager method that CASes Running→Stopped, kills the tree, enqueues PendingItem::Stopped{reason: StoppedByUser} (single-writer preserved), fires roster-changed; a ClientFrame::KillMonitor{id} dispatch mirroring handle_abort. StoppedByUser already exists, unused.
  2. Event sink — Phase B (full unification). Make the sink the sole emission path: the loop emits via sink.emit() too; WS/tests observe the sink, not the run-stream. Cures the “emission coupled to the loop” smell. Main cost = migrating how the test harness observes events (Agent::send_message collects the returned stream). Deliberately gated on whether Phase A earns its keep.
  3. Action-vs-effect completeness (schema-wide policy). The log is inconsistent about recording the action vs the effect. Sharpest case: monitor stdout — queue time (action) is computed as InputQueueItem.enqueued_at for the live queue-viz but discarded at delivery; only MonitorDeliveryEvent.time (effect) is persisted. Inspection step: enumerate every OmegaEvent variant, classify action / effect / both, decide a documented rule. Low cost, high conceptual payoff.
  4. Retire / reposition wait_for_output. Once monitors are proven the better path: remove it, drop it from the default selection, or keep + note monitors are preferred. Evidence-driven.
  5. Reconsider auto-stop of chatty monitors. Dropped by explorative minimalism; kept by Claude Code. Re-add on evidence of the agent leaking/flooding, or keep observing.
  6. Per-tool toggle for non-UI sessions. A --tool / --no-tool CLI flag (or config) so headless/scripted sessions toggle individual tools without a custom preset; today only coarse --preset exists server-side.
  7. One-shot vs streaming tool split (design discussion). Claude routes “notify me once” to a background job and reserves Monitor for streaming; Omega collapses both into one monitor. Revisit splitting them + a default timeout_ms / persistent opt-out. Observe first.
  8. Live queue/badge polish. Sub-seam real-time queue streaming + per-line badge updates. Seam-aligned already suffices — low-value polish.
  9. Monitors controllable inside the Python REPL. Expose the monitor queue as a Python API so REPL code reacts to events in-session. Speculative; the tool-call boundary is the right abstraction for now.
