1. Goal & motivation

Omega today has wait_for_output, which is synchronous: it occupies the single tool-call slot and blocks the agent loop for the whole wait. We want an asynchronous monitor primitive instead — the agent registers a condition and keeps working (or parks), and when the condition fires, a message is injected as a user message at the next valid seam (boundary). This is the Claude Code "Monitor" model. No preemption — monitors never interrupt an in-flight message; they wait for the next boundary.

Target use cases (all expressible as one shell-command primitive):

  • Wait a duration — sleep N; echo done.
  • Wait for a process to finish — wait $pid; echo done.
  • Arbitrary boolean condition — a poll-and-if loop that prints only when the condition flips.

3. The seam rules (Anthropic Messages API — correctness, not opinion)

Terminology (read this first — “turn” is banned for being overloaded)

  • message — one API object: a single { role, content } where role is user or assistant. The atomic unit. We say user message / assistant message, never “user turn.” Anthropic combines consecutive same-role messages into one server-side.
  • cycle — one agent loop iteration: a single human (or monitor) prompt that drives possibly many assistant + tool-result messages until the assistant emits end_turn. In Omega code this is what send_message runs, finishing at TurnEnd. A cycle is made of many messages.
  • end_turn — Anthropic’s stop_reason on an assistant message meaning “done, yielding control.” A property of one message, not a unit of conversation.
  • boundary (a.k.a. seam) — a point where a user message may legally be inserted: after end_turn (Seam A) or immediately after a completed tool_result (Seam B). Defined precisely below.

Monitors deliver at a boundary, as a user message, never preempting an in-flight message or splitting a tool_use/tool_result pair.

  • Two roles only: user / assistant. Anything not generated by the model — tool results, hook output, monitor notifications, human input — must enter as role:user. "Impersonating the user" is forced by the protocol, not a design choice; both Claude and opencode do it. The synthetic flag is only a harness/UI marker to distinguish it from a human-typed message.
  • tool_use → tool_result is rigid: every tool_use block must be immediately followed, in the very next message, by matching tool_result blocks. You cannot insert a separate message between them.
  • Exactly two legal injection points for a user message:
    • Seam A — after the assistant ends with end_turn and is idle → start a fresh user message.
    • Seam B — piggybacked after the tool_result blocks in the user message that answers a tool_use.
  • A "boundary" = end_turn OR a completed tool_result. "Generating" and "ToolRunning" are merely between boundaries. A generation always resolves into one of the two boundary kinds; if it resolves to a tool_use, you reach Seam B once the tool returns.
  • No preemption at the context level. The only way to act before the next natural boundary is to abort the in-flight request (transport level) — that "manufactures a seam now." We keep abort as a separate, explicit interrupt path; it is not how monitors deliver.
  • A long/blocking tool call holds Seam B shut for its whole duration → starvation. This is precisely the wait_for_output problem, and the reason monitors must be an out-of-band channel rather than a blocking tool.

4. Core mechanic: pull-at-boundary (single writer)

  • Monitors run as background OS processes. Each stdout line is enqueued to an in-memory queue. Monitors never write events.jsonl.
  • The main loop is the single writer. It drains the queue only at the two boundaries:
    1. right after it appends a tool_result, before calling the model again; and
    2. when it is about to park at end_turn.
  • When parked, the loop blocks on a select over three wake sources: human input and monitor output (both arrive in the queue), plus roster-changed — a signal that the set of live monitors mutated without producing any queue output (a monitor was killed or exited on its own). Any of the three wakes the loop; it then drains the queue and re-evaluates park-vs-terminate.
  • No-preemption is free: the loop simply never inspects the queue between boundaries.
  • The monitor subsystem is dumb — it knows nothing about the fold or the tool_use/tool_result invariant. That ignorance is the safety; only the loop, which is always at a legal boundary when it drains, ever appends.

What “roster” means

The roster is the running process’s in-memory list of currently-live monitors — pure ephemeral runtime state, never persisted and never reconstructed. events.jsonl records MonitorStarted/MonitorStopped for forensics, but the live set itself is held only in RAM; after a restart it is empty (see §10, Lifecycle). Each entry carries id, description, command, status, start time, event-fired count, and a stderr tail — exactly what the UI badge/modal renders (§9) and what the WS layer pushes ephemerally (§7).

Why roster-changed is a wake reason. The park/terminate rule is “park iff a live monitor or queued event exists; otherwise terminate.” When the agent is parked on its last live monitor and that monitor is killed (UI KillMonitor) or exits by itself, no queue line is produced, so neither of the other two wake sources fires. Without a roster-changed wake the loop would sit parked forever even though the terminate condition is now satisfied. The signal forces a re-evaluation so the loop exits cleanly instead of hanging — it is the mechanism behind “never hang when nothing can ever fire.”

─────────────────────          ─────────────────────────        ────────────────
monitor procs --stdout line--> [ pending queue ] --drain at----> events.jsonl

Phase 1 build note — queue/roster ownership & clean cutover

Where the queue and roster live. Both sit together in a single MonitorManager (crates/omega-tools/src/monitors.rs) held behind an Arc and stored on ToolCtx.monitors: Option<Arc<MonitorManager>> — mirroring how the python_repl state is threaded onto ToolCtx. This is the smallest ownership model that bridges Phase 1 → Phase 2: the monitor/stop_monitor tools reach it now through the ctx handle, and Phase 2’s main loop will hold the same Arc, clone it into every ToolCtx it builds, and call drain_pending() at each boundary plus shutdown() on session end — no re-architecting. It is deliberately not a module-level static (unlike the run_background registry): a global would leak the roster across sessions and across tests, violating the ephemeral-roster rule (§10) and making tests non-deterministic.

Clean cutover — no half-wired sessions. A session must be either unaware of monitors or all-in, never in between. Phase 1 ships the runtime and the two tools but does not wire the loop drain, so exposing the tools to the model now would let it call monitor() and spawn a process whose output nothing ever drains — a half-wired state. Therefore Phase 1 keeps the tools invisible: monitor/stop_monitor are not in ALL_TOOL_NAMES / DEFAULT_TOOL_NAMES / any preset, and tool_definitions never emits their schemas, so the model can never select or call them (the agent also rejects any tool name outside ALL_TOOL_NAMES). The execute_tool dispatch arms exist purely so the end-to-end tests can drive the tools directly. The cutover is a single, atomic, reviewable change in a later phase: add the two names to the registry and emit their schemas, together with the loop drain. Until then, real sessions are byte-for-byte unaffected.

Phase 1 — as built (API surface the Phase 2 loop wires against)

Everything below lives in crates/omega-tools/src/monitors.rs (runtime) plus tools/monitor.rs / tools/stop_monitor.rs (the two tools). Commit e101dfb. Internally the manager is one Mutex<ManagerInner> holding the roster (HashMap<id, entry>), the pending queue (VecDeque<PendingItem>), and an id counter. 

// Shared handle: agent owns one Arc, clones it into every ToolCtx.
MonitorManager::new() -> Arc<Self>

// Spawn `bash -c command` in its own process group (pgid = child pid via
// process_group(0)), piped stdout/stderr, null stdin, kill_on_drop. Starts
// 3 tokio tasks (stdout reader, stderr reader, waiter) and RETURNS IMMEDIATELY.
spawn(self: &Arc<Self>, description, command) -> Result<SpawnedMonitor{id, started_at}, String>

stop(&self, id) -> bool        // kills whole tree (kill_group); true iff it was live (caller logs
                               //   MonitorStopped); false = no-op for unknown / already-stopped
shutdown(&self) -> Vec<String> // session-end reaping: kill all live trees, return their ids
drain_pending(&self) -> Vec<PendingItem>   // Phase 2 calls this AT THE TWO BOUNDARIES
roster(&self) -> Vec<MonitorInfo>         // ephemeral snapshot (UI/WS, Phase 3)
status(&self, id) -> Option<MonitorStatus>

enum PendingItem { Stdout{monitor_id, line}, Stderr{monitor_id, chunk} }
enum MonitorStatus { Running, Stopped }
struct MonitorInfo { id, description, command, status, started_at, fired_count, stderr_tail }
const STDERR_TAIL_MAX = 20;   // roster tail is bounded; full stderr still flows through the queue

Event emission = single writer, via extra_events. The tools do not append events. monitor() returns MonitorStarted and stop_monitor() returns MonitorStopped{reason: StoppedByAgent} on the ToolResult.extra_events vec; the agent’s dispatch loop persists those (same mechanism as python_repl’s PythonReplBootstrapped). Phase 0’s Agent::append_monitor_started / inject_monitor_delivery / inject_monitor_stopped remain unused for now — they are the hooks for the loop-driven paths Phase 2 adds (delivery on drain; ProcessExited/ProcessCrashed stops).

Phase 2 wiring checklist (no re-architecting needed):
  1. Create one Arc<MonitorManager> per session in the agent; clone it into each ToolCtx (replace the Phase 1 monitors: None in agent.rs).
  2. At the two boundaries call drain_pending() and map items: Stdout → MonitorDelivery (batch per monitor; projected role: user), Stderr → MonitorStderr (non-projected). Reuse inject_monitor_delivery.
  3. Parking select over {human, queue, roster-changed}; fire roster-changed when a live monitor exits/killed without queue output.
  4. Call shutdown() on session end (reaps trees — no orphans).
  5. Cutover (atomic): add monitor/stop_monitor to ALL_TOOL_NAMES (+ default/preset), add their schema fns so tool_definitions emits them, and add the deferred teaching copy (§5).
Phase 2 decisions (as built) — ✔:
  • Natural-exit kill_group: YES. When the waiter sees a monitor self-exit (Running→Stopped), it calls kill_group(pgid) before enqueuing the stop item. A self-exited bash leader can leave background grandchildren reparented to init; because they inherit the leader’s process group, one group kill reaps them. Cost: an already-dead leader makes the kill a harmless no-op. The alternative (reap only the leader) was rejected — it silently leaks the exact orphans the process-group design exists to prevent. The SIGKILL the waiter observes for an agent-killed monitor is suppressed (the entry is already Stopped), so this never double-classifies as ProcessCrashed.
  • DEFAULT / preset: opt-in, not default. monitor and stop_monitor join ALL_TOOL_NAMES and the all preset only — not DEFAULT_TOOL_NAMES/standard (same posture as python_repl). Rationale: monitors are an advanced async capability whose correct use depends on the teaching copy being present (§5 teaching-copy prerequisite); making them opt-in keeps the default session lean and ensures a monitor-aware session is deliberately selected. Explorative minimalism (§5): no dedicated new preset — all already means “everything”.

Deliberate Phase 1/2 boundary (not an oversight). In Phase 1 the waiter task only reaped the child and flipped the roster entry to Stopped — it did not capture the exit code, distinguish ProcessExited vs ProcessCrashed, or enqueue a stop item, so projected stop delivery for self-terminating monitors was unwired. Phase 2 wired this (see the Phase 2 entry in §11): the waiter now classifies the ExitStatus and enqueues PendingItem::Stopped, and on the natural transition also kill_groups for orphan reaping. (Also: stderr’s roster cap uses a single if, not a while — safe because pushes are one-at-a-time, and it keeps the boundary mutation catchable rather than an infinite-loop timeout.)

5. Guardrail philosophy

Explorative minimalism. Add only guardrails that are obviously load-bearing for correctness. Deliberately skip the rails Anthropic added to protect a broad public. Iterate on agent features by observing the single agent's behavior, not by pre-empting every footgun. Omega's Web UI makes this safe: human-in-the-loop observability substitutes for automated kill.

KEEP — obviously load-bearing (correctness)

  1. Never inject between tool_use and its tool_result — enforced structurally by pull-at-boundary (no decision to get wrong).
  2. Single writer — only the main loop appends events; monitors enqueue.
  3. Distinct event variant for monitor injections (not a reused human user message) — per the AGENTS.md loud-schema-change rule. Projects to role:user.
  4. Identity + lifecycle + stop — each monitor has an id; start/fired/stopped attributable; agent can stop by id; stopping a dead monitor is a no-op.
  5. Process-tree reaping on stop and on session end (no orphans).
  6. Defined park/terminate condition — park iff a live monitor or queued events exist; otherwise terminate. Never hang when nothing can ever fire.

DROP — public-serving rails (skip now, re-add on evidence)

RailWhy we skip it
Default + hard-max timeout_msProtects strangers from forgotten monitors. We have explicit stop + session-end reaping. No timeout default; observe whether the agent leaks monitors first.
Auto-stop chatty monitors (firehose)Public-protective. We are one human watching one agent. UI count + activity is the backstop. Observe whether the agent self-corrects.
"Always use grep --line-buffered" / poll-interval naggingPrompt nannying. Ship the raw primitive; add guidance only if observation shows laggy/buggy monitors.

Tool surface

Ship the raw, general primitive first: monitor(description, command) — stdout lines are events; returns immediately with a monitor id. Plus stop_monitor(id). No sugar (monitor_until/_for/_process), no timeout, no auto-stop, no baked-in filter advice. Add a typed helper only if observation shows the agent repeatedly fumbles the shell boilerplate.

Prerequisite: the prompt and tool descriptions must teach monitors

Explorative minimalism (§5) drops prompt nannying — micro-advice like “always use grep --line-buffered.” It does not drop the baseline conceptual explanation the feature needs to be usable at all. Before monitors ship, the system prompt and the monitor/stop_monitor tool descriptions must clearly convey:

  • What a monitor is — an async, out-of-band channel; monitor() returns immediately and stdout lines arrive later as injected user messages.
  • The seam guarantee — no preemption; a monitor never interrupts an in-flight message, so the agent can rely on atomic cycles.
  • When to use which — monitors vs the synchronous wait_for_output (see the open decision in §12); the agent should not reach for the blocking tool when an async wait will do.
  • Stop discipline — monitors must be stopped with stop_monitor(id) when no longer needed (session-end reaping is a backstop, not licence to leak).

This is load-bearing for usability, distinct from the public-serving rails we drop. Treat it as a deliverable alongside the tools (Phase 1) and verify with a MockProvider end-to-end test that the agent reaches for monitors where appropriate.

6. Signed-off decisions

#Decision
1No auto-stop. Runaway monitors are caught by visibility, not automation. Visible in both logs and UI. The UI is what earns the right to skip auto-stop.
2Batch at seam. Everything pending is delivered together at the next boundary. Canonical events stay distinct (per source); the context projection merges consecutive user-role content into one API message.
3stderr → events.jsonl (forensics paramount), but excluded from the context projection. New event class: diagnostic / non-projected — in the canonical log + UI, never in the model context (no token cost, never wakes the agent). Accept a chatty log; coalesce later; no stable event format needed yet.
4Unify the pending queue. Monitor lines and human input flow through one queue. Guiding principle: "when I (the human) queue a message, I am a human monitor." Items tagged by source (human / monitor-id). Default = enqueue (delivered at next seam, no preempt). Explicit interrupt (abort) stays a separate path (Enter = enqueue; explicit interrupt = abort/preempt the in-flight cycle).
5Visualize the pending queue in the UI, including human messages.
6Unexpected/killed terminations surface into context (agent must learn so it does not wait forever). Clean agent-initiated stops need not.
7Collapsed-badge first enhancement = events-fired/activity (the runaway detector), ahead of timing stats.
8Monitor liveness is runtime-only, never reconstructed from the log. But forensics is fully in the log.
9Clean shutdown kills all monitors + best-effort writes MonitorStopped. Hard crash leaves a dangling MonitorStarted — that is intended forensic signal, not a bug.
10Resume forks a new session (see code facts); no live-monitor inheritance; nothing to reconcile.
11Headless = benchmark-only, bounded by an external time limit + session-end cleanup. Residual token-burn risk from a runaway monitor = observe, don't guard.

7. Layer mapping (Contract Authority — types are the schema)

Contract rule (from AGENTS.md): the Rust OmegaEvent in crates/omega-types/src/events.rs is the canonical schema. Persistence, WS protocol, and UI are derived projections. A semantic change must get a syntactic change (new variant), not a silent alias. Update the UI to match the types — never the reverse.
LayerWhat lives here
Canonical OmegaEvent MonitorStarted, MonitorDelivery (the batched synthetic message, projects role:user), MonitorStderr (diagnostic, non-projected), MonitorStopped. See §8.
Persistence events.jsonl = all events. context.jsonl = derived projection: a per-record log (each record keyed by its own ContextHash) that excludes MonitorStderr. (events.jsonl ⊋ context.jsonl.) The merge of consecutive user-role records into one API message is a separate request-assembly step — project_messages() in crates/omega-agent/src/agent.rs, applied when the LlmRequest is built, not when context.jsonl is written. The per-record log is left un-merged so the ContextHash foreign keys stay stable.

8. Proposed event schema (sketch — finalize in events.rs first)

Field names illustrative; the point is the variant boundaries:

// canonical OmegaEvent variants (camelCase on the wire)
MonitorStarted { id, description, command, time }
MonitorDelivery {                       // drained at a seam; projects role:user
    time,
    items: [ { source, lines } ]        // source = Human | Monitor(id); per-source attribution
}                                        // (human messages MAY stay their existing user-message
                                         //  event, drained alongside — see open item)
MonitorStderr  { id, chunk, time }       // DIAGNOSTIC: in events.jsonl, NOT in context.jsonl
MonitorStopped { id, reason, exit_code, time }
                                         // reason = StoppedByAgent | StoppedByUser | ProcessExited | ProcessCrashed | StoppedBySessionEnd
  • The context fold must skip MonitorStderr.
  • MonitorDelivery (and any human user message drained with it) projects to role:user; consecutive user-role events merge into one API message at the projection boundary (satisfies role alternation + "batch at seam").
  • Unexpected MonitorStopped (StoppedByUser/ProcessCrashed/ ProcessExited while the agent depends on it) should be projected into context so the agent learns; StoppedByAgent need not (agent already knows it called stop_monitor), and StoppedBySessionEnd need not (session is terminating — no loop to notify).

10. Lifecycle, shutdown, resume, headless

  • Within a running session: monitors are live runtime entities in the in-memory roster.
  • Clean shutdown: kill all monitor process trees + best-effort write MonitorStopped for forensics.
  • Hard crash / kill -9: dangling MonitorStarted with no Stopped; a forensic reader infers "process died here." Acceptable — intended signal.
  • Resume (handle_resume_session): forks a new session from a resumption basis; does not inherit live monitors. Nothing to reconcile. Verify nothing tries to revive monitors from the prior log.
  • WS reconnect/replay (should_replay): re-sends logged events to render the view; live roster reflects whatever the running process currently holds (empty after a restart). Liveness is never reconstructed.
  • Headless: benchmark-only, bounded by an external time limit + session-end cleanup. No human-kill needed there.

E2E tests via Agent::send_message+MockProvider; mutation recipes mutants-agent-projection, mutants-monitors, mutants-system-prompt, mutants-schemas (all caught/unviable). Deviation from the original sketch: human input is not yet routed through the unified queue — it remains the existing per-message send_message path, with the parked loop woken via the ControlHandle (continue = interjection, abort = terminate). Routing free-typed input through the queue and surfacing the parked state in the UI is folded into Phase 3; the three logical wake sources all exist today.
  4. Phase 3 — WS + UI. ✔ DONE (read-only roster surface).

    Scope: server-authoritative ephemeral roster push. WsMessage::MonitorRoster variant added to the transport layer (never an OmegaEvent; never persisted to events.jsonl). The server stashes Arc<MonitorManager> in ActiveSession and pushes a fresh snapshot (a) on client connect/reconnect, (b) after reset/resume, and (c) after each MonitorStarted/MonitorDelivery/MonitorStderr/MonitorStopped event in the streaming task. Frontend: roster: RwSignal in SessionStore; MonitorsBadge (header badge, hidden when empty, shows running count + total fired count) + MonitorsModal (roster table with all fields) in frontends/leptos/src/monitors_panel.rs. Tests: WS-level tests (roster sent on connect-with-session; roster not in events.jsonl), unit tests for is_monitor_event and roster_snapshot_msg, snapshot tests for badge/modal. Mutation recipes: mutants-monitor-roster-push (3 mutants, 2 caught, 1 unviable), mutants-monitor-ws-message (0 mutants — serde_json::json! macros are not mutated; wire-format covered by unit tests that snapshot the exact JSON output).

    Deferred (see §9): Kill button / KillMonitor control frame; live pending-queue visualization; real-time per-line badge updates.

  5. Phase 4 — Shutdown / resume / headless. ✔ DONE.

    Scope: (a) MonitorStopReason rename + new variant; (b) shutdown logging; (c) resume-safety proof; (d) headless CLI teardown.

    (a) MonitorStopReason rename + StoppedBySessionEnd. The original variant names (AgentStopped, UserKilled, Crashed) were renamed to StoppedByAgent, StoppedByUser, ProcessCrashed to disambiguate actor vs event. A new variant StoppedBySessionEnd was added (wire value: stopped_by_session_end). Per the loud-schema-change rule: the teardown path is a different actor (session lifecycle) from an explicit stop_monitor call (StoppedByAgent) — they must not share a variant. Old logs with the previous wire values (agent_stopped, user_killed, crashed) fail loudly on deserialization (no defensive #[serde(alias)]) as intended. should_project() returns false for both StoppedByAgent (agent already knows) and StoppedBySessionEnd (session is terminating, no loop to notify); true for StoppedByUser, ProcessExited, ProcessCrashed.

    (b) Shutdown logging. Agent::shutdown_and_log_monitors() kills every live monitor's process tree (via the existing MonitorManager::shutdown() CAS) and persists a MonitorStopped(StoppedBySessionEnd) event for each. Single-writer preserved: the server calls request_abort() first, ensuring the agent loop has stopped writing before teardown writes. Wired at omega-server/src/router.rs (handle_reset and handle_resume_session, before the slot is replaced) and crates/omega-cli/src/main.rs (after the stream drains, before returning the exit code). Agent::drop remains a kill-only backstop (no logging — Drop is synchronous and has no access to the event store).

    (c) Resume-safety proof. A resumed session always gets a fresh empty MonitorManager; handle_resume_session only replays prior events into LLM context, never calls monitor_start or spawns processes. Two tests in omega-server/src/router.rs prove this: resume_does_not_revive_completed_monitor (prior session has MonitorStarted+MonitorStopped) and resume_does_not_revive_dangling_monitor (prior session has only MonitorStarted, simulating a crash before clean shutdown). Both assert an empty roster after resume.

    (d) Headless CLI teardown. Tests in crates/omega-agent/tests/internal.rs: shutdown_and_log_monitors_kills_process_and_persists_session_end_stop (process killed + event logged) and shutdown_and_log_monitors_does_not_double_log_already_stopped_monitor (CAS guard prevents double-logging for naturally-exited monitors).

12. Single-source-of-truth refactor & monitor promotion (implemented)

This was an unscheduled interlude refactor between Phases 3 and 4 — NOT Phase 4 itself. It completed the following structural changes:

  • Single source of truth — tool registry moved to crates/omega-types/src/tools.rs. Previously, tool-name lists (DEFAULT_TOOL_NAMES, ALL_TOOL_NAMES, REPL_CENTRIC_TOOLS), the Preset struct, PRESETS, preset_by_id, and the pure selection helpers (default_tool_selection, resolve_preset, serialize_selection, parse_stored_selection) existed in three places: omega-tools/src/schemas.rs, frontends/leptos/src/protocol.rs, and frontends/leptos/src/picker.rs. They are now in omega-types/src/tools.rs only, with pub use re-exports in the other two files so existing call sites continue to compile. Structural drift (like the Phase-2 bug where monitors were added to the backend but not the frontend) is now structurally impossible — there is exactly one list.
  • Monitors promoted to default across all three presets. DEFAULT_TOOL_NAMES grew from 12 to 14 tools (monitor and stop_monitor added). REPL_CENTRIC_TOOLS grew from 3 to 5 tools (same additions). ALL_TOOL_NAMES stays at 15 (= default 14 + python_repl). All three presets now include monitor tools: Standard (14), All (15), REPL-centric (5).
  • Off-switch: Monitors category in the UI picker. A new “Monitors” category was added to TOOL_CATEGORIES in frontends/leptos/src/picker.rs, exposing monitor and stop_monitor as per-tool checkboxes. Users can deselect them to create a Custom preset without monitors.
  • Consistency guard test. A suite of structural invariant tests was added to crates/omega-types/src/tools.rs: every preset’s tools are a subset of ALL_TOOL_NAMES; ALL_TOOL_NAMES equals DEFAULT_TOOL_NAMES plus python_repl; both monitor tools are present in all three presets; every picker category tool is known. These tests would have caught the original Phase-2 drift at CI time.

13. Open items (deferred, non-blocking)

  • stderr coalescing granularity — by-lines vs by-time. Decided: accept chatty log for now, optimize later; no stable format required.
  • Human-in-delivery representation — does MonitorDelivery aggregate human + monitor items in one event, or does the human message stay its existing user-message event drained alongside a monitor delivery? Lean: keep them as separate canonical events; the projection merges them. (Either way the UI tags by source.)
  • Exact reason enum for MonitorStopped and which reasons project into context (lean: all "unexpected" ones do).
  • Does MonitorStarted re-surface into context? Lean: no — the tool result already informs the agent; the event is for log/causality.
  • Retire / reposition wait_for_output? Monitors are the async replacement for the synchronous, Seam-B-blocking wait_for_output (§1, the original motivation). Do not remove it at Phase 0 — that would delete a working capability before its replacement exists, and Phase 0 is schema-only. Options once monitors are proven: (a) hard-remove the tool; (b) keep it but drop it from the default tool selection; (c) keep it and note in its description that monitors are preferred for non-blocking waits. Decision deferred until we observe whether the agent still reaches for the blocking tool with monitors available. (Closely tied to the prompt/tool-description prerequisite in §5.)
  • PLANNED: General per-tool toggle mechanism (CLI/non-UI). Today the only server-side mechanism to change the active tool set is --preset <id> (three coarse presets). Per-tool toggling is currently UI-only (the picker checkboxes). A future --tool <name> / --no-tool <name> CLI flag — or a config-file equivalent — would allow headless/scripted sessions to selectively enable or disable individual tools without writing a custom preset. Out of scope for now.
  • OPEN QUESTION: Should monitors be controllable from inside the Python REPL? The REPL-centric preset includes monitor and stop_monitor; the agent can call them as normal tools. A deeper integration would expose the monitor queue as a Python API inside the REPL (e.g., omega.start_monitor(...)) so that REPL code can react to monitor events within the same Python session. This is architecturally interesting but out of scope — the tool-call boundary is the correct level of abstraction for now.
